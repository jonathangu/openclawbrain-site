Benchmark + operator proof

Proof, with the boundary stated plainly

by Jonathan Gu

Short version: on three long-lived-memory benchmark families, OpenClawBrain beat the strongest RAG-style baseline by +9.7 to +26.9 percentage points, won 29 of 30 seed-level matchups, and used 5x less context per query. One family is recorded replay on fixed fixtures; two are deterministic seeded simulations. Separately, install, attach, and restart are now verified on real OpenClaw profiles. That is strong benchmark evidence and real operator proof, but it is not yet a claim of live learning on real profiles.

The big picture Operator proof Setup guide Operator guide

Real-profile install/attach/restart is verified; live learning on attached profiles remains a separate proof boundary. See the Setup Guide, Operator Guide, and CLAIMS.md for the exact boundary.

Benchmark headline

29-1-0 across three families

  • Relational drift: recorded replay, 99.15% vs 89.44%, +9.71 pp, 10-0-0
  • Recurring workflows: seeded simulation, 97.6% vs 70.6%, +26.9 pp, 10-0-0
  • Sparse feedback: seeded simulation, 92.0% vs 67.0%, +24.9 pp, 9-1-0

10 seeds per family. 8 baselines in each family. 5x less context throughout. Full artifacts live in brain-ground-zero.

Operationally verified

Install, attach, restart

npm install -g @openclawbrain/openclaw@0.3.0 worked on real OpenClaw profiles. The default install now expects an embedder and provisions bge-large before brain bootstrap unless the operator explicitly uses --skip-embedder-provision or OPENCLAWBRAIN_INSTALL_SKIP_EMBEDDER_PROVISION=1. The local Ollama teacher path is still optional. Attach succeeded on two real profiles, and a Telegram-connected profile came back cleanly after restart.

This is proof of the public install and restart surface. It is not yet proof of live learning after export and promotion.

Read this like a reviewer

What to keep in mind while reading

  • The benchmark claim is narrow: better long-lived memory on these three task families.
  • Relational drift is the strongest evidence because it is recorded replay, not simulation.
  • The sparse-feedback family keeps its one seed loss visible; we do not round that into a perfect sweep.

Read this like a skeptical technical reviewer

Start with the claim, the measurement setup, and the places where the evidence is weaker. This page keeps those separate on purpose.

1 · Claim

What is actually being claimed

OpenClawBrain's memory stack beats the strongest RAG-style baseline on three long-lived-memory benchmark families. This is not a general "better assistant" claim.

2 · Measurement

How the evidence is split

One family is recorded replay with deterministic fixture verification. Two families are deterministic seeded simulations. All three run at 10-seed scale and report seed-level outcomes.

3 · Boundary

What you should not infer

No claim yet on live LLM routing, production latency, production cost, larger world sizes, or live learning after export on attached profiles. Those remain separate future proofs.

Family 1: Relational drift

Facts about organizations change over time. The system has to track who reports to whom, which team owns which service, and when old answers become wrong. This is the strongest evidence in the package because it is recorded replay with deterministic verification.

Recorded replay · 10 seeds · 800 queries/seed

99.15% vs 89.44%

  • Main result: +9.71 pp margin and a 10-0-0 seed record.
  • Context use: 5x lower (800 vs 4,000 tokens per query).
  • Comparison target: the strongest RAG baseline in this family, vector_rag_rerank.
  • Why this matters: deterministic fixture replay with SHA-256 verification means the run is fully reproducible and not dependent on simulator behavior.
  • Coverage: 8 baselines total, including oracle, RAG variants, and partial versions of the full system.

Public bundle: proof-results/recorded_h2h_relational_drift_10seed.

Family 2: Recurring workflows

Repeated tasks like deploy checklists, review protocols, and onboarding flows should get easier as the system builds memory. This family shows the largest benchmark advantage: +26.9 percentage points.

Seeded simulation · 10 seeds · 3,522 queries/seed

97.6% vs 70.6%

  • Main result: +26.9 pp margin and a 10-0-0 seed record.
  • Context use: 5x lower.
  • Evaluation type: deterministic seeded simulation, not live traffic replay.
  • This family has the largest query count in the package: 3,522 queries per seed.

Public bundle: proof-results/recurring_workflows_10seed.

Family 3: Sparse feedback

Real teams do not correct every answer. This family asks whether the system still improves when only about 19% of queries receive explicit feedback. The result is strong, but the caveat matters.

Seeded simulation · 10 seeds · 1,800 queries/seed

92.0% vs 67.0%

  • Main result: +24.9 pp margin and a 9-1-0 seed record.
  • Context use: 5x lower.
  • Evaluation type: deterministic seeded simulation with sparse correction events.
  • Important caveat: the ±18.3% standard deviation on full_brain means seed outcomes vary a lot when feedback is rare. Which turns get corrected matters. One seed loss means we do not claim a universal sweep.

Public bundle: proof-results/sparse_feedback_10seed.

What this proves — and what it doesn't

This is the shortest honest version of the claim boundary. If you only read one section after the hero, read this one.

Supported now

The memory system beats RAG on these benchmark tasks

  • The full system beats the best RAG-style baseline and all measured partial-system ablations on three distinct long-lived-memory benchmark families.
  • Each family runs at 10-seed scale with 8 baselines. Results are deterministic given a seed.
  • Memory structure updates are the single largest contributor in every family; the ablation ladder does not reverse.
  • Context efficiency: 5x lower across all three families.
Do not infer this yet

Where the claim still stops

  • Performance on memory_compaction (designed but not run).
  • Behavior at larger world sizes (current: 50 entities / 80 workflows).
  • End-to-end performance with live LLM routing (harness uses simulated policy functions).
  • Production latency or cost.
  • Live learning proof on attached real profiles after export, promotion, and later served turns.
  • Real-session head-to-head on product traces (spec defined, no scored results).
  • "Always wins" across all families (sparse feedback is 9-1-0).

Operator proof: what is real in the field

This section is intentionally separate from the benchmarks. It answers a different question: does the shipped install and restart path work on real OpenClaw profiles? Yes. Has live learning on those profiles been proven yet? No.

1 · Benchmark proof

Benchmark evidence is strong

  • Recorded H2H exists for relational drift: deterministic replay, SHA-256 verified, 10-0-0.
  • Recurring workflows and sparse feedback extend coverage to two more long-lived-memory families.
  • The benchmark claim is superiority over RAG-style baselines on these tasks, not live-runtime learning proof.
2 · Install / attach / restart proof

Real OpenClaw profiles now attach cleanly under the shipped install contract

  • npm install -g @openclawbrain/openclaw@0.3.0 worked on a real operator path.
  • Normal install now expects an embedder and provisions or pulls bge-large before brain bootstrap by default.
  • Operators can explicitly skip that step with --skip-embedder-provision or OPENCLAWBRAIN_INSTALL_SKIP_EMBEDDER_PROVISION=1 if they manage the embedder separately.
  • The local Ollama teacher path remains optional.
  • The safest default remains one dedicated activation root per profile.
  • Attach succeeded on two real OpenClaw profiles.
  • A Telegram-connected profile came back cleanly after the verified restart path.
env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway restart
3 · Live learning proof

Live learning proof remains separate

  • Install and attach prove the operator surface, not that export, learning, and promotion have already happened live.
  • Install and attach do not by themselves prove that exported turns have refreshed the live profile.
  • The next honest proof boundary is exported turns landing, then later served turns reflecting promoted knowledge.
Install surface

To reproduce the verified path, start with the Setup Guide. Use the Operator Guide for status, rollback, and attach triage. Why this matters

Trust the single turn, then trust the aggregate

  • You can inspect one turn end-to-end: what context was sent, what answer came back, what correction was captured.
  • The benchmark scores are meaningful because every row traces back to the same per-turn contract.
  • Corrections do not update the guide immediately. New knowledge takes effect on the next background cycle, not in the same served turn.
