Evidence package

Proof package.

RAG retrieves chunks by similarity and hopes the right answer is in the pile. OpenClawBrain replaces that with a learned router — trained on real usage, it walks a knowledge graph, picks the right context, and improves from corrections. On two frozen benchmark suites, the learned router scores 97.5% vs. 89.6% for the best RAG baseline, and 92.0% vs. 67.1% under sparse feedback — using roughly 5× less context. These are controlled benchmark results, not live production claims.

Latest benchmark results

What still needs testing

Shadow-mode and live-traffic evidence are next

  • For recorded-session claims: we publish the exact queries, state snapshot, and scoring rubric used across all modes so anyone can verify.
  • For sparse-feedback claims: we publish per-seed results plus context size and graph traversal budgets for each mode.
  • For shadow-mode claims (not yet complete): we will mirror real OpenClaw traffic, compare brain-on vs brain-off outputs, and save every disagreement.
  • For live rollout claims (not yet complete): we will publish success rates, correction counts, prompt sizes, latency, and cost from actual served traffic.
How we label claims

Every claim on this site gets one of three labels

  • Proven now: the evidence exists, you can download it, and you can rerun the test yourself.
  • Requires more evidence: we believe this is true but need shadow-mode or live-traffic tests to back it up.
  • Not claimed: we do not have the evidence yet, so we do not make the claim.

Why this matters

Trust the single turn before trusting the scores

  • Proves now: you can inspect one turn end-to-end — context sent, answer returned, correction captured.
  • Connects to benchmarks: aggregate scores below are meaningful because every row traces to this same per-turn contract.
  • Does not prove: live production answer quality, shadow-mode outcomes, or live rollout success.
  • Learning boundary: corrections do not update the router immediately. New weights take effect on the next background cycle.

Workflow proof: does the right context reach the model?

For 4 fixed workflow queries, does the router surface exactly the right knowledge-graph nodes? Vector search: 0 of 4. Graph structure alone: 2 of 4. Learned router: all 4 — combining structural edges with learned weights. Learning runs in the background and never slows a live response.

What this tells us

Works out of the box, gets better with learning

  • Even before any learning, the graph structure alone gets the right context for 2 of 4 queries -- this is the "cold start" baseline.
  • The detailed per-query matrix (per_query_matrix.csv) shows exactly which knowledge-graph nodes reached the model's context for each query and retrieval mode.
  • After background learning (which runs off the live path), the router gets all 4 queries right by epoch 14 with 100% coverage of required nodes.
  • Simple link-following (pointer_chase) averages 1.25 hops and still only gets 1 out of 4 right -- structure alone is not enough.

This matrix proves which knowledge-graph nodes reached the model's context in a controlled simulation. It does not prove downstream answer quality, real-session task success, or production latency and cost on live OpenClaw traffic.

Recorded-session head-to-head

800 frozen queries, locked knowledge state, one rubric for every method. The learned router answers correctly about 8 pp more often than the best RAG baseline — using a fraction of the context. Benchmark evidence, not a live production claim.

Benchmark result

Learned router: 97.5% correct. Best RAG baseline: 89.6%.

  • 800 frozen queries, 41 training steps, 8 retrieval methods compared side by side.
  • Learned router (full_brain) accuracy: 97.5%
  • Best RAG baseline (vector_rag_rerank): 89.6%
  • Gap: +7.9 percentage points -- roughly 1 in 12 queries where the learned router is right and RAG is wrong.

Public bundle: proof-results/recorded_h2h_relational_drift_001 in brain-ground-zero at commit 1c54302. Run ID: recorded_h2h_relational_drift_001.

Proof boundary

What this does and does not prove

  • This is a controlled benchmark: frozen queries, locked state, same scoring rubric for every method.
  • It proves the learned router is more accurate than 8 conventional retrieval methods on this workload.
  • It does not prove live production answer quality, shadow-mode wins, or live rollout outcomes.
  • Next steps: shadow-mode testing (mirroring real traffic) and narrow live rollout.

Sparse-feedback proof (10 random seeds)

Real teams do not correct every answer. This benchmark tests the harder scenario: does the learned router still win with sparse corrections? Across 10 independent seeds the advantage widens to nearly 25 pp — because learned routing compounds even from limited feedback, while RAG cannot improve at all.

Benchmark result

Learned router: 92.0% correct. Best RAG baseline: 67.1%.

  • 10 independent random seeds, each with sparse (minimal) correction signals.
  • Learned router (full_brain) average accuracy: 91.96%
  • Best RAG baseline average accuracy: 67.05%
  • Gap: +24.91 percentage points -- the learned router gets roughly 1 in 4 additional queries right.
  • Efficiency: the learned router sends roughly 5x less context to the model per query, meaning faster and cheaper responses.
  • Head-to-head: the learned router wins 9 out of 10 seeds against the best RAG baseline.

Public bundle: proof-results/sparse_feedback_10seed in brain-ground-zero (run date 2026-03-07). Reproduce entrypoint: ./scripts/run_sparse_feedback_proof.sh.

Proof boundary

What this does and does not prove

  • This proves the learned router is significantly more accurate and more efficient than RAG baselines even when correction signals are sparse.
  • It shows the advantage holds across 10 independent random seeds, not just one lucky run.
  • It does not prove live production answer quality, real-world latency gains, or live rollout outcomes.
  • Shadow-mode testing and narrow live rollout are the next required evidence steps for those claims.

This rung is reached. This is still controlled benchmark evidence, not live production data.

Shadow mode

Does it work on real traffic? The brain runs alongside the live system without serving answers. We compare outputs, save every disagreement, and measure prompt size, latency, and cost.

This step builds confidence before we start serving live answers. Not yet complete.

Does it improve real outcomes for real users? The brain serves a small slice of live traffic with automatic fallback. Claims are published only when backed by saved data from that slice.

This is the real product proof -- not simulation, not benchmark. Not yet complete.

Artifact checklist

Every claim on this site has a matching evidence bundle. These are the files that back each claim type.

Worked turn

One traceable turn

  • Stable turn ID and input summary
  • Exact routing parameters used
  • The bounded context block sent to the model
  • The response or answer text
  • Any correction event and its learning outcome
Simulation run

Mechanism test bundle

  • Exact command and config to reproduce
  • Code commit and random seed
  • Output directory with all generated files
  • Charts (SVG/PNG) and data files (JSON/CSV)
  • Figure paths referenced on this site
Head-to-head

Benchmark comparison bundles

  • Query set, its checksum, and seed list
  • Knowledge-state snapshot identifier
  • Scoring rubric version and per-method outputs
  • Per-query correctness judgments
  • Per-seed summary with context size budgets
  • Sampled worked-turn bundles for spot-checking
Shadow or live

Real-traffic comparison data

  • Traffic slice and date range
  • Brain-off vs brain-on outputs for same queries
  • Prompt size, latency, and cost logs
  • Saved disagreement traces (where outputs differed)
  • Rollout window and automatic fallback policy
