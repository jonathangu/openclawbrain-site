What proof is for

Use proof when you need durable evidence or a shareable bundle, not because it should be the only health check.

  • bundle verdict: success_and_proven means install, runtime, breadcrumbs, load proof, and detailed status aligned in the captured bundle.
  • When a fresh host looks healthy but proof still warns, read status --detailed first and keep the bundle for later comparison.
  • The bundle writes summary.md, verdict.json, steps.json, logs, breadcrumbs, and runtime-load proof into one directory.
  • Pack ids and counters are host-state dependent. The healthy signals matter more than exact numbers.
Replay reality check: checked replay already beats no_brain on real traces, and learned_route has real-trace wins. The replay set is still small and mixed, so the broad winner is not settled and direct spend savings are not yet proven.

Canonical commands

Install, restart, verify, prove

openclawbrain install --openclaw-home '~/.openclaw'
openclaw gateway restart
openclawbrain status --openclaw-home '~/.openclaw' --detailed
openclawbrain proof --openclaw-home '~/.openclaw'
How to read it

Proof is the durable record, not the first read

If status --detailed already shows the healthy signals, proof should mostly confirm and package them, even if it still emits warning-level noise on a fresh host.
