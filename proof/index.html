Benchmark evidence

Proof package.

by Jonathan Gu

The learned router scores 97.5% correct vs 89.6% for the best RAG baseline on 800 recorded-session queries. Under sparse feedback (10 seeds), the gap widens to 92.0% vs 67.1% with roughly 5x less context per query. This page collects all the evidence, explains exactly what it proves, and draws a clear line where the claims stop.

These are benchmark results, not live-traffic claims. Shadow-mode and online rollout evidence are the next steps. See CLAIMS.md for the authoritative boundary.

What is proven today

Two kinds of evidence, both published

  • Automated integration tests (this repo): verify the promoted-pack lifecycle, learned routing function, and operator observability. Run pnpm lifecycle:smoke, pnpm observability:smoke, pnpm fresh-env:smoke.
  • Comparative benchmarks (brain-ground-zero repo): recorded-session and sparse-feedback test suites comparing the learned router against 8 baselines on frozen workloads.

Recorded-session head-to-head

800 frozen queries, locked knowledge state, one rubric for every method. The learned router answers correctly about 8 pp more often than the best RAG baseline while using a fraction of the context.

Benchmark result

Learned router: 97.5%. Best RAG: 89.6%.

  • 800 frozen queries, 41 training steps, 8 retrieval methods compared side by side.
  • Gap: +7.9 percentage points — roughly 1 in 12 queries where the learned router is right and RAG is wrong.
  • The learned router uses roughly 5x less context per query than vector_rag_rerank.

Public bundle: proof-results/recorded_h2h_relational_drift_001 in brain-ground-zero at commit 1c54302.

Proof boundary

What this does and does not prove

  • Proves the learned router is more accurate than 8 conventional retrieval methods on this frozen workload.
  • Does not prove live production answer quality, shadow-mode outcomes, or live rollout results.
  • Next steps: shadow-mode testing (mirroring real traffic) and narrow live rollout.

Sparse-feedback proof (10 random seeds)

Real teams do not correct every answer. This benchmark tests the harder scenario: does the learned router still win with sparse corrections? The advantage widens because the router compounds learning from even limited feedback, while RAG has no feedback loop at all.

Benchmark result

Learned router: 92.0%. Best RAG: 67.1%.

  • 10 independent random seeds, each with sparse correction signals.
  • Gap: +24.9 percentage points.
  • Roughly 5x less context per query.
  • Wins 9 out of 10 seeds head-to-head.

Public bundle: proof-results/sparse_feedback_10seed in brain-ground-zero.

Proof boundary

What this does and does not prove

  • Proves the learned router is more accurate and more efficient than RAG baselines even with sparse corrections, across 10 independent seeds.
  • Does not prove live production answer quality, real-world latency gains, or live rollout outcomes.

Worked-turn example

Before trusting aggregate scores, inspect one concrete turn. This shows what happens when OpenClaw asks the brain for context: the routing decision, the bounded context sent to the model, and how a correction ties back to the exact route that fired.

Why this matters

Trust the single turn before trusting the scores

  • You can inspect one turn end-to-end: context sent, answer returned, correction captured.
  • Aggregate benchmark scores are meaningful because every row traces to the same per-turn contract.
  • Corrections do not update the router immediately — new weights take effect on the next background cycle.

Workflow proof: does the right context reach the model?

4 fixed queries test whether the router surfaces the exact right knowledge-graph nodes. Vector search: 0/4. Graph structure alone: 2/4. Learned router: 4/4. Learning runs in the background and never slows a live response.

What this tells us

Useful on day one, better with learning

  • Graph structure alone gets 2/4 right before any learning — the "cold start" baseline.
  • After background learning (off the live path), the router gets all 4 right by epoch 14.
  • Simple link-following (pointer_chase) gets only 1/4 — structure alone is not enough.

This proves which knowledge-graph nodes reached the model's context in a controlled simulation. It does not prove downstream answer quality on live traffic.
