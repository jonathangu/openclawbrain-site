What this page proves, and what it does not

This page is intentionally narrow. It proves install, load, serving, and reporting truth on a healthy exercised host surface, then shows how to reproduce that comparison on yours.

  • bundle verdict: success_and_proven means the captured bundle aligned across install, runtime, plugin load, breadcrumbs, runtime-load proof, and detailed status.
  • STATUS ok, loadProof=status_probe_ready, serve state=serving_active_pack, and routeFn available=yes are the top-level public truth signals.
  • proof writes summary.md, steps.json, verdict.json, raw step logs, startup breadcrumbs, and the runtime-load-proof snapshot into one durable bundle.
Boundary: this page does not claim every host will have the same pack id, graph size, route count, or supervision count. Those vary by host state. It also does not claim universal answer-quality gains by itself. It claims that the install lane can be verified honestly.

Run the same lane on your host

Keep the exact same selected home on the OpenClawBrain commands before and after restart. Compare the fast status surface first. Capture proof second.

Canonical commands

Install, restart, verify, prove

openclawbrain install --openclaw-home '/Users/guclaw/.openclaw'
openclaw gateway restart
openclawbrain status --openclaw-home '/Users/guclaw/.openclaw' --detailed
openclawbrain proof --openclaw-home '/Users/guclaw/.openclaw'
Why the path stays pinned

One selected home stays in scope

install, status, and proof all accept --openclaw-home. Keep them pinned to /Users/guclaw/.openclaw. openclaw gateway restart does not take that flag, so the restart step is the only command in the lane without it.

If you explicitly manage @openclawbrain/openclaw with the OpenClaw plugin manager, rerun openclawbrain install --openclaw-home '/Users/guclaw/.openclaw' afterward so the hook and activation root converge again.
