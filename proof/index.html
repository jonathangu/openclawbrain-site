Proof

Don’t trust memory. Inspect it.

A memory system should leave evidence. OpenClawBrain exposes the load state, SQLite health, proof events, graph, search, route learning, and recent injection decisions.

The point is not to claim perfect memory. The point is to make memory visible enough that you can debug it.

Start with runtime truth

openclaw plugins inspect openclawbrain --runtime
openclaw doctor

You want the plugin loaded, hooks registered, runtime routes present, and SQLite + FTS healthy.

A package can be installed without being useful. Runtime inspect tells you whether the plugin is actually alive inside the gateway.

Useful surfaces

SurfaceWhat it answers
/plugins/openclawbrain/statusIs memory enabled? Which mode is active? What happened recently?
/plugins/openclawbrain/doctorCan the runtime open SQLite, use FTS, and write proof rows?
/plugins/openclawbrain/proof?limit=20What did the plugin decide, inject, skip, store, or reject?
/plugins/openclawbrain/search?query=pnpm&limit=20Can local memory search find the thing you taught it?
/plugins/openclawbrain/graph?agentId=main&limit=50What redacted memory nodes and edges exist for this agent?
/plugins/openclawbrain/learn?limit=50What route examples and learned policy snapshots exist?
/plugins/openclawbrain/audit?limit=20Which capture attempts were stored, skipped, or rejected?
/plugins/openclawbrain/explain-lastWhat happened on the latest memory decision?

HTTP plugin routes are authenticated on normal installs. Use the authenticated OpenClaw dashboard/client, or pass your gateway auth header when using curl.

End-to-end proof shape

A useful smoke test is small:

  1. Teach a correction: Use pnpm instead of npm in this repo.
  2. Confirm it appears in search or graph.
  3. Ask for a test/build task.
  4. Confirm a bounded memory block was injected.
  5. Check proof or explain-last for the decision.

That is the core product claim: capture a durable lesson, retrieve it later, inject it inside budget, and leave proof behind.

What good proof looks like

Load proof

The plugin is enabled, loaded, and registered as an OpenClaw memory plugin.

Storage proof

SQLite opens, migrations are current, FTS works, and proof rows can be written.

Recall proof

A known memory can be found through local search and selected for a relevant turn.

Injection proof

The prompt receives a small, scoped memory block rather than a big transcript dump.

Learning proof

Route examples, outcomes, and policy snapshots update as the system sees more turns.

Safety proof

Rejected captures, redaction, scoping, and skipped injections are visible enough to debug.

The claim, kept narrow

OpenClawBrain does not make agents magically perfect. It does something more useful: it gives OpenClaw a local, inspectable way to remember feedback and use it at the right time.

  • Memory stays local by default.
  • The LLM proposes semantic updates.
  • Code validates and writes.
  • SQLite stores the graph and evidence.
  • Prompt injection stays sparse and bounded.

If a memory system cannot show you what it did, it is too hard to trust. OpenClawBrain is built so you can look.
