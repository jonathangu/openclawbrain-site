Benchmark evidence

Does it actually work?

by Jonathan Gu

Think of OpenClawBrain as a librarian who learns which shelves matter for each question — instead of keyword-searching every book. On 800 recorded queries, this librarian answers 97.5% correctly vs 89.6% for the best search-everything approach. When corrections are scarce (10 random seeds), the gap widens to 92.0% vs 67.1% with roughly 5x less material pulled per question. This page collects the evidence, explains what it proves, and draws a clear line where the claims stop.

These are benchmark results, not live-traffic claims. Testing against real user traffic is the next step. See CLAIMS.md for the authoritative boundary.

What we can show today

Two kinds of evidence, both public

  • Integration tests (this repo): verify the full lifecycle — building a knowledge snapshot, activating it, and confirming the guide picks the right context. Run pnpm lifecycle:smoke, pnpm observability:smoke, pnpm fresh-env:smoke.
  • Head-to-head benchmarks (brain-ground-zero repo): the learned guide vs 8 conventional search methods on frozen workloads, scored by the same rubric.

800 recorded queries, same rules for everyone

Every method gets the same frozen queries, the same knowledge state, and the same scoring rubric. The learned guide answers correctly about 8 percentage points more often than the best search-based method, while pulling a fraction of the material.

The numbers

Learned guide: 97.5%. Best search method: 89.6%.

  • 800 frozen queries, 41 training steps, 8 retrieval methods compared side by side.
  • Gap: +7.9 percentage points — roughly 1 in 12 queries where the guide is right and search gets it wrong.
  • The guide uses roughly 5x less context per query than the best search method.

Public bundle: proof-results/recorded_h2h_relational_drift_001 in brain-ground-zero at commit 1c54302.

Where the claim stops

What this does and doesn't prove

  • Proves the learned guide is more accurate than 8 conventional retrieval methods on this frozen workload.
  • Does not prove live production quality, real-traffic outcomes, or live rollout results. Those come next.
  • Next steps: shadow-mode testing (mirroring real traffic) and narrow live rollout.

What happens when corrections are rare

Real teams don't correct every answer. This test checks the harder scenario: can the guide still win when it only occasionally hears "that was wrong"? The advantage actually widens, because the guide compounds learning from even limited feedback. Search methods have no feedback loop at all — they perform the same on day one and day one thousand.

The numbers

Learned guide: 92.0%. Best search: 67.1%.

  • 10 independent random seeds, each with sparse correction signals.
  • Gap: +24.9 percentage points.
  • Roughly 5x less context per query.
  • Wins 9 out of 10 seeds head-to-head.

Public bundle: proof-results/sparse_feedback_10seed in brain-ground-zero.

Where the claim stops

What this does and doesn't prove

  • Proves the learned guide is more accurate and efficient than search methods even with sparse corrections, across 10 independent seeds.
  • Does not prove live production quality, real-world latency gains, or live rollout outcomes.

Why this matters

Trust the single turn, then trust the aggregate

  • You can inspect one turn end-to-end: what context was sent, what answer came back, what correction was captured.
  • The benchmark scores are meaningful because every row traces back to the same per-turn contract.
  • Corrections don't update the guide immediately — new knowledge takes effect on the next background cycle, like a librarian updating the card catalog overnight.

Does the right knowledge actually reach the model?

4 fixed queries test whether the guide surfaces exactly the right pieces of knowledge. Keyword search: 0/4. The knowledge map alone: 2/4. The learned guide: 4/4. Think of it like asking a new librarian vs an experienced one — the experienced one knows which facts connect to your question, not just which words match.

What this tells us

Useful on day one, better with practice

  • The knowledge map alone gets 2/4 right before any learning — that's the "useful from the start" baseline.
  • After background learning (which never slows a live response), the guide gets all 4 right by round 14.
  • Simple link-following gets only 1/4 — following obvious connections isn't enough without learning which connections matter most.

This proves which pieces of knowledge reached the model in a controlled test. It does not prove downstream answer quality on live traffic.

Integration tests you can run yourself

These tests run entirely inside the OpenClawBrain TypeScript workspace with no external dependencies. They verify the full lifecycle: building a knowledge snapshot, activating it, and confirming the guide picks the right context.

pnpm lifecycle:smoke

Full lifecycle end-to-end

Events flow in, a candidate snapshot is built, activated, promoted — and the guide serves from the promoted snapshot with verified checksums.

pnpm observability:smoke

Operator visibility

Health checks, freshness, rollback history, label counts, route diffs, and fallback usage — everything an operator needs to see what the guide is doing.

pnpm fresh-env:smoke

Clean install from zero

Install from local packages into an isolated directory. Attach, promotion, and rollback proved end-to-end from a blank slate.

pnpm continuous-product-loop:smoke

The ongoing learning loop

Events export, supervision, first serve, candidate refresh, promotion, and later served-turn verification — the whole cycle the guide runs through continuously.
