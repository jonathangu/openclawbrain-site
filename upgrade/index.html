OpenClawBrain upgrade / repair

Rerun the same install lane on an existing host.

Upgrade is not a second hidden workflow. When a host needs package refresh, hook repair, or boundary reconciliation, rerun the same pinned install lane and then restart and verify.

That keeps the product honest: one front door for a new host and the same front door for an existing one that needs repair.

What upgrade means here

The job is to get an existing host back into the same pinned, inspectable state as a clean install.

Refresh packages

Pick up the current published split

The live pair is @openclawbrain/openclaw@0.4.5 and @openclawbrain/cli@0.4.16.

Repair drift

Re-converge the hook and activation root

The install lane repairs the host boundary without making you learn a new command family.

Verify again

Restart, then read status before proof

If proof is still noisy after the rerun, detailed status is the quicker way to confirm the runtime is healthy.

Canonical upgrade lane

Use the same command you would use for a new install. The difference is the reason you are running it, not the command surface. 

openclawbrain install --openclaw-home '~/.openclaw'
openclaw gateway restart
openclawbrain status --openclaw-home '~/.openclaw' --detailed
openclawbrain proof --openclaw-home '~/.openclaw'
Upgrade truth: the published sterile dogfood lane still had warning-level proof noise, even though the healthy runtime signals were already present. That is why the upgrade page emphasizes restart and detailed status, not proof alone.

When this page is the right page

Package drift

Your installed host is older than the published pair

Rerun the install lane so the host picks up the current runtime and CLI packages.

Hook drift

The plugin or activation root needs repair

Use this page when the boundary is present but no longer converged cleanly.

Post-change verify

You want to confirm the runtime actually loaded

Restart and detailed status tell you whether the live path is healthy before you bother with proof.

Go deeper

If the host still looks wrong after the rerun, go straight to troubleshooting or compare the healthy reference proof.
