Setup Guide

This guide walks you through installing OpenClawBrain, attaching it to an OpenClaw profile, and confirming it is working. Most operators can go from zero to a running brain in a few minutes.

OpenClaw owns the live runtime (agent orchestration, tool calls, responses). OpenClawBrain owns the knowledge layer (memory, learning, and context compilation). You install OpenClawBrain separately, then attach one or more OpenClaw profiles to the brain without adding response-path latency.

What you need

  • Node.js 20+
  • A running OpenClaw deployment — OpenClawBrain is a learning layer, not a standalone runtime
  • Workspace files (markdown, docs, memory files) for the agent to learn from
No API keys needed

OpenClawBrain uses local BGE-large embeddings and a local Ollama/Qwen teacher by default. No cloud provider keys are required for the brain itself. Your OpenClaw runtime may still need its own provider keys for agent and tool calls — see Secrets and Capabilities.

1 Install

Install the published package globally, then confirm the CLI is available:

npm install -g @openclawbrain/openclaw@0.3.0
openclawbrain --help

Start with openclawbrain install. After attach, use the status and rollback tooling to inspect health or back out a bad pack if needed.

2 Attach an OpenClaw profile

Run the installer for the profile you want to wire up. A shared activation root per machine is a valid default: multiple OpenClaw profiles can point at the same brain, and retrieval can still differ per ask because context compilation happens at request time. Choose a dedicated activation root only when you want stricter isolation for rollback, cleanup, or blame.

openclawbrain install \
  --openclaw-home "$PROFILE_HOME" \
  --activation-root "$ACTIVATION_ROOT" \
  --workspace-id "$PROFILE_NAME"
Shared by default, isolated on purpose

Reusing one activation root for the machine is acceptable. Give a profile its own root when you want operator actions such as rollback, disable, or cleanup to stay scoped to that profile alone.

The installer provisions a local bge-large embedder by default. To skip embedder provisioning (if you already have one), pass --skip-embedder-provision or set OPENCLAWBRAIN_INSTALL_SKIP_EMBEDDER_PROVISION=1.

3 Restart and verify

Restart the OpenClaw gateway for your profile, then check status:

env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway restart
env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway status

After restarting, verify these behaviors:

  1. Agent responds immediately. OpenClaw boots from the seed pack without waiting for background learning. A fresh brain with zero events stays in seed state until the first exported turn arrives — this is normal.
  2. Background learning starts. New events are exported first; historical backfill continues in the background. The learner builds candidate packs off the hot path; activation promotes them when ready.
  3. Fail-open works. If brain workers are delayed, OpenClaw still serves from the last active pack. Missing packs fail open so responses continue. However, a learned-required route artifact that drifts will hard-fail instead of silently serving stale data.

4 Check health

If you are working from the source repo, these helper scripts give you a quick read on the activation root you installed against. With a shared root they report machine-level brain state; with a dedicated root they report just that profile's brain state.

pnpm operator:status     # one-screen summary: did attach land, what is active, is rollback ready?
pnpm operator:doctor     # PASS / WARN / FAIL follow-up when status shows drift

Rollback and recovery

If something goes wrong after a pack promotion:

  • Brain unavailable: OpenClaw automatically falls back to core runtime behavior. Responses continue without brain-enhanced context.
  • Bad brain pack: Preview the rollback first, then apply it:
# Preview the pointer move (no changes made)
pnpm operator:rollback -- --activation-root <path> --dry-run

# Apply the rollback
pnpm operator:rollback -- --activation-root <path>

Use the same --activation-root you installed against. With a shared root, rollback moves the shared machine brain back for every attached profile using that root; with a dedicated root, rollback stays isolated to that profile. Compilation from a given pack is deterministic, so rollback is safe and repeatable. Learning delays do not affect the hot path — scanner, harvest, and learner workers all run asynchronously.

Runtime configuration

The install step wires most of this automatically. If you need to inspect or customize the runtime profile, here is the shape OpenClaw expects:

{
  "brain": {
    "enabled": true,
    "packSet": "<versioned-pack-set>",
    "fastBootFromExistingFiles": true,
    "backgroundLearning": {
      "enabled": true,
      "prioritizeNewEvents": true
    },
    "labels": { "human": true, "self": true },
    "scanner": { "enabled": true },
    "harvest": { "enabled": true },
    "teacher": { "onHotPath": false }
  }
}

Key points about the current behavior:

  • fastBootFromExistingFiles — OpenClaw starts serving immediately from existing files; bootstrap attach works even from zero events.
  • teacher.onHotPath: false — learning never adds latency to responses.
  • Labels, scanner, and harvest feed into route training. They run on the learning side, not in the serve loop.
  • Background learning means: export turns, build a candidate pack off the hot path, then promote. Eagle and Tern now prove that passive-learning loop on attached installs. The active pack is still not mutated in-place during the same served turn.
What is not here yet

Scanner, labels, and harvest are not yet on by default after attach — you still configure them explicitly. Eagle/Tern do prove graph evolution through export, learning, and promoted-pack change on attached installs. What is still not claimed is same-turn in-place mutation of the currently active pack during serving, or broader rollout hardening.

Working with the source repository (maintainers)

If you are contributing to OpenClawBrain itself:

corepack enable
pnpm install --frozen-lockfile
pnpm check                  # type-check, lint, test, verify package integrity
pnpm release:check          # full proof lane + local tarballs
pnpm fresh-env:smoke        # clean-room outside-consumer install proof

A brain pack is an immutable versioned artifact containing graph nodes, vector entries, provenance, and an optional learned route function. It is built off the hot path by the learner and made visible only after promotion by activation. The compiler always serves from the currently active promoted pack.

Proof lanes

Run these to verify the full promoted-pack lifecycle locally:

pnpm lifecycle:smoke                   # normalized events → promoted-pack compilation
pnpm observability:smoke               # activation health, PG metadata, route artifact diffs
pnpm continuous-product-loop:smoke     # live-style export → candidate refresh → promotion
pnpm runtime-wrapper-path:smoke        # OpenClaw wrapper: compile → export → learn → promotion
pnpm customer-value-proof:smoke        # recorded-session replay showing learning improves later turns
