Setup Guide

This is the day-zero path: install OpenClawBrain, attach it to an OpenClaw profile, and make sure the brain is actually breathing. Most operators can get from zero to a live attach in a few minutes.

OpenClaw runs the live agent loop: orchestration, tool calls, and responses. OpenClawBrain handles memory, learning, and context compilation behind it. You install the brain separately, then attach one or more OpenClaw profiles without adding response-path latency.

What you need

  • Node.js 20+
  • A running OpenClaw deployment — OpenClawBrain is a learning layer, not a standalone runtime
  • Workspace files (markdown, docs, memory files) for the agent to learn from
No API keys needed

OpenClawBrain keeps its own work local: BGE-large embeddings plus an optional Ollama/Qwen teacher path. No cloud-provider keys are required for the brain itself. Your OpenClaw runtime may still need provider keys for agent and tool calls — see Secrets and Capabilities.

1 Install

Install the published package globally, then make sure the CLI answers back:

npm install -g @openclawbrain/openclaw@0.3.0
openclawbrain --help

Start with openclawbrain install. After attach, use the status and rollback tools to check whether the brain landed cleanly and to back out a bad pack if needed.

2 Attach an OpenClaw profile

Run the installer for the profile you want to wire up. A shared activation root per machine is an acceptable default: multiple OpenClaw profiles can point at the same brain, and retrieval can still differ per ask because context compilation happens at request time. Choose a dedicated activation root when you want rollback, disable, cleanup, or uninstall scoped to one profile.

openclawbrain install \
  --openclaw-home "$PROFILE_HOME" \
  --activation-root "$ACTIVATION_ROOT" \
  --workspace-id "$PROFILE_NAME"
Shared by default, isolated on purpose

Reusing one activation root for the machine is acceptable. Give a profile its own root when you want operator actions such as rollback, disable, cleanup, or uninstall to stay scoped to that profile alone.

The installer provisions a local bge-large embedder by default. If you already manage your own embedder, skip that step with --skip-embedder-provision or OPENCLAWBRAIN_INSTALL_SKIP_EMBEDDER_PROVISION=1.

3 EXTERNAL RESTART and verify

Restart the OpenClaw gateway externally for that profile, then ask it for status:

env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway restart
env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway status

After restarting, verify these behaviors:

  1. Agent responds immediately. OpenClaw boots from the seed pack without waiting for background learning. A fresh brain with zero events stays in seed state until the first exported turn arrives. That is normal.
  2. Background learning starts. New events are exported first; older history can backfill in the background. The learner builds candidate packs off the hot path, and activation promotes them when they are ready.
  3. Fail-open works. If brain workers are delayed, OpenClaw still serves from the last active pack. Missing packs fail open so responses continue. A learned-required route artifact that drifts will hard-fail instead of quietly serving stale data.

4 Check health

If you are working from the source repo, these helper scripts give you the first operator check-in surfaces on the activation root you installed against. With a shared root they report machine-level brain state; with a dedicated root they report just that profile's brain state.

pnpm operator:status     # one-screen summary: did attach land, what is active, is rollback ready?
pnpm operator:doctor     # PASS / WARN / FAIL follow-up when status shows drift

Rollback and recovery

If something goes wrong after a pack promotion:

  • Brain unavailable: OpenClaw falls back to core runtime behavior. Responses keep flowing, just without brain-enhanced context.
  • Bad brain pack: Preview the rollback first, then apply it:
# Preview the pointer move (no changes made)
pnpm operator:rollback -- --activation-root <path> --dry-run

# Apply the rollback
pnpm operator:rollback -- --activation-root <path>

Use the same --activation-root you installed against. With a shared root, rollback moves the shared machine brain back for every attached profile using that root; with a dedicated root, rollback, disable, cleanup, or uninstall stay isolated to that profile. Compilation from a given pack is deterministic, so rollback is safe and repeatable. Learning delays do not touch the hot path; scanner, harvest, and learner workers all run asynchronously.

Runtime configuration

The install step wires most of this automatically. If you want to inspect or customize the runtime profile, this is the shape OpenClaw expects:

{
  "brain": {
    "enabled": true,
    "packSet": "<versioned-pack-set>",
    "fastBootFromExistingFiles": true,
    "backgroundLearning": {
      "enabled": true,
      "prioritizeNewEvents": true
    },
    "labels": { "human": true, "self": true },
    "scanner": { "enabled": true },
    "harvest": { "enabled": true },
    "teacher": { "onHotPath": false }
  }
}

Key points about the current behavior:

  • fastBootFromExistingFiles — OpenClaw starts serving immediately from existing files; bootstrap attach works even from zero events.
  • teacher.onHotPath: false — learning never adds latency to responses.
  • Labels, scanner, and harvest feed into route training. They run on the learning side, not in the serve loop.
  • Background learning means: export turns, build a candidate pack off the hot path, then promote. Eagle and Tern now prove that passive-learning loop on attached installs, including learned route_fn refresh and graph truth updates on later served turns. The active pack is still not mutated in-place during the same served turn.
What is not here yet

Scanner, labels, and harvest are not yet on by default after attach, so you still configure them explicitly. Eagle/Tern do prove graph evolution through export, learning, and promoted-pack change on attached installs. What is still not claimed is same-turn mutation of the currently active pack during serving, or broader rollout hardening.

Working with the source repository (maintainers)

If you are contributing to OpenClawBrain itself:

corepack enable
pnpm install --frozen-lockfile
pnpm check                  # type-check, lint, test, verify package integrity
pnpm release:check          # full proof lane + local tarballs
pnpm fresh-env:smoke        # clean-room outside-consumer install proof

A brain pack is an immutable versioned artifact containing graph nodes, vector entries, provenance, and an optional learned route function. The learner builds it off the hot path, activation promotes it, and the compiler serves only from the currently active promoted pack.

Proof lanes

Run these to verify the full promoted-pack lifecycle locally:

pnpm lifecycle:smoke                   # normalized events → promoted-pack compilation
pnpm observability:smoke               # activation health, PG metadata, route artifact diffs
pnpm continuous-product-loop:smoke     # live-style export → candidate refresh → promotion
pnpm runtime-wrapper-path:smoke        # OpenClaw wrapper: compile → export → learn → promotion
pnpm customer-value-proof:smoke        # recorded-session replay showing learning improves later turns
