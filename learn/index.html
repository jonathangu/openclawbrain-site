How OpenClawBrain Works

Your agent starts every conversation with amnesia. Here’s why that happens, and exactly how OpenClawBrain fixes it — from first principles.

Install safely See it in action See the evidence

The problem: every conversation starts from scratch

Most agents rely on RAG — retrieval-augmented generation. When you ask a question, the system searches a pile of notes for text that looks similar to your question, then stuffs that text into the prompt.

It sounds reasonable. In practice, it breaks in predictable ways:

  • Keyword overlap does not mean conceptual relevance. A question about “memory” retrieves notes about RAM, childhood, and your API docs — in no particular order.
  • Corrections are forgotten. You tell the agent it got something wrong. Next session, it makes the same mistake again. No one wrote the correction into the index in a useful way.
  • Stale context never fades. An architectural decision from six months ago is weighted equally with something you changed yesterday.
  • There is no map of what connects to what. The system retrieves fragments, not understanding.

The solution: a trained guide, not a search engine

OpenClawBrain replaces keyword search with something closer to how an expert colleague works. Instead of scanning text for matching words, it maintains a structured map of your knowledge — facts and rules connected by relationships — and uses a learned routing function to pick what’s actually relevant to the current situation.

When you correct the agent, that correction updates the map. The wrong path weakens; the right path strengthens. Over time, the agent gets better at your specific domain, not just better at sounding fluent.

And if OpenClawBrain is unavailable for any reason, your agent keeps working. It falls back to baseline context rather than failing. The system is designed to help, never to block.

Operator bridge: understand the mechanism, then install from the right boundary

The install path is real now, but it is still an operator path. After you understand how the brain works, the next step is to attach it from a normal shell or deployment flow, not from a live chat window.

Safe default today: one activation root per profile, then restart the runtime externally so the profile comes back with the brain attached. Install and restart are real. Live learning proof starts later, after exports and promotion.

How to install it safely

  • Run install and attach from a normal shell, script, or deploy lane, not from the live chat itself.
  • Use one dedicated brain activation root per profile by default. It keeps state, rollback, and blame clear.
  • Restart OpenClaw externally after install so the profile reloads with the attached brain cleanly.
  • Use the site setup guide for the day-0 path, then the operator runbook for checks and rollback.

What is proven today

Verified install onto real profiles now exists. Attach plus external restart succeeded for Tern and CormorantAI, and Tern came back alive on Telegram after restart.

That is install proof, not full live-learning proof. Both verified brains are still waiting on first export, so no one should read install success as proof that later turns are already learning from fresh live data.

Keep the claim boundary straight: a successful install proves the profile can carry a brain safely. It does not by itself prove that the learner has exported, promoted a fresher pack, and changed later behavior. That evidence belongs on the proof path, not in the install claim.

Key concepts

Four ideas underpin how OpenClawBrain works.

☉

Knowledge Graph

Facts and rules are stored as nodes connected by typed edges — not as a flat pile of text chunks. A node might be a rule your team follows; an edge connects it to the context where that rule applies. Retrieval follows the map, not word overlap.

▶

Learned Routing

A trained function decides which parts of the graph are relevant to the current conversation. It learns from what was useful and what was not — so retrieval improves with use. It is not looking for similar-sounding text; it is predicting what context will actually help.

↻

Structural Plasticity

The map evolves. Connections that get used and confirmed grow stronger. Stale knowledge that is never recalled fades over time. Corrections reshape the graph immediately — not in a future batch update.

▶│

Fail-Open Design

OpenClawBrain runs behind OpenClaw. If the brain is slow or unavailable, OpenClaw proceeds with baseline context. Your agent never hard-fails because of the memory layer. The worst case is the same as no memory layer at all.

OpenClaw vs OpenClawBrain

They work together, but they do different things. Understanding the split makes the system much easier to reason about.

Runtime

OpenClaw

The front desk
  • Runs the agent and manages conversations
  • Handles orchestration and tool dispatch
  • Routes requests to OpenClawBrain for context
  • Proceeds with baseline context if brain is unavailable
  • Fails independently of the memory layer

Memory & Learning

OpenClawBrain

The back office
  • Maintains the knowledge graph
  • Selects context using learned routing
  • Applies corrections and updates the graph
  • Strengthens active connections, fades stale ones
  • Fails independently of the runtime

No cloud required. Both components run locally. The brain stores everything on disk in your environment — nothing is sent to a third-party service. For live profiles today, the safe default is one activation root per profile, with attach and restart handled outside the conversation itself.
