Persistent brain for OpenClaw

One brain per machine. Better context per ask. Corrections that keep paying off.

OpenClawBrain gives OpenClaw a persistent brain that learns off the hot path, retrieves the right context for each ask, and improves recurring workflows over time. The intended default is simple: one shared local brain per machine is acceptable, dedicated roots stay available when you want tighter isolation, and OpenClaw remains fail-open.

What this changes on a real task

A simple example: an agent keeps missing the exact rollback rule during a deploy incident. The difference is not abstract. It shows up on the next similar task.

Before OpenClawBrain

The agent repeats the same retrieval mistake

You correct the bad answer once, but the next incident still pulls a broad runbook, misses the precise rollback policy, and spends tokens on context that does not help.

After OpenClawBrain

The correction shows up when the task comes back

The correction is stored as reusable memory, linked to the relevant policy, and retrieved with a smaller prompt. On the next similar incident, the agent starts closer to the right rollback path instead of relearning it from scratch.

What changes in practice

OpenClaw still runs the agent. OpenClawBrain changes what the agent remembers and what context it sees next time. In day-to-day use, that shows up in three ways.

Fewer repeated mistakes

Corrections become reusable memory

When you fix a mistake, OpenClawBrain records what went wrong and why. The next time a similar situation comes up, the agent can retrieve that correction instead of repeating the error.

Per-ask retrieval

The brain does not show the same memory every time

A trained routing function picks the context that matters for this ask instead of dumping in every vaguely related chunk. Prompts get smaller, and two similar questions can still retrieve different memory when the task actually differs.

Better recurring workflows

Repeated workflows improve

Outcomes and feedback feed back into training. Workflows your agent runs repeatedly get better over time as the routing function learns what context actually helps.

How it works

Three product ideas, in plain English.

Knowledge graph

Facts live as connections, not loose documents

Instead of a pile of text chunks, knowledge lives in a graph. "This deploy policy depends on that rollback rule" becomes a real edge, not two documents that just happen to share a word.

Learned routing

A trained router decides what this ask needs

When a question arrives, the router walks the graph and selects the most useful nodes for that ask. It learns from corrections and outcomes, so it gets better at picking the right context over time.

Structural plasticity

The memory graph reshapes as reality changes

Nodes connect, split, merge, and prune under policy control. Stale knowledge fades, active connections strengthen, and ablations show this is the largest single accuracy contributor across every benchmark family.

Current status

Public technical alpha — stronger operator truth, same ambitious target

OpenClawBrain is in early alpha. The product direction is one shared brain per machine, with retrieval decided per ask and learning happening in the background, while dedicated roots remain available when you want tighter isolation. The benchmark story is still ahead of the production-hardening story, but the public operator proof is now real: Eagle verified install, passive learner start, first export, promotion, learned route_fn, default bge-large, optional qwen3.5:9b teacher enablement, and graph updates showing up on later served turns, while Tern verified attach and clean restart on a Telegram-connected profile. That is narrow live operator proof, not broad production hardening or live answer-quality proof. Expect breaking changes before a stable release. Apache-2.0 licensed. Requires Node 20+.

Read the split literally. Benchmark proof says the memory architecture beats strong RAG baselines. Operator proof now says the install, passive-learning, and served-pack story is real on live machines. Broad same-gateway many-profile safety and shared-root concurrency guarantees are still separate proof work.

Install for OpenClaw operators

If you want to try it now, use the published package and run openclawbrain install from a normal shell. The current front door provisions bge-large by default, can enable a compatible local qwen3.5:9b teacher, and can start passive learning before the next gateway restart.

Current honest read: Eagle proved install can auto-start the exact-root passive learner, reach first export, and serve from a promoted pack with live bge-large embeddings and an enabled local qwen3.5:9b teacher. Tern proved the attach-and-restart path on a live Telegram-connected profile. One shared brain per machine is acceptable when you want one memory surface; dedicated roots still buy clearer isolation, rollback, and blame. Skip embedder provision only with --skip-embedder-provision or OPENCLAWBRAIN_INSTALL_SKIP_EMBEDDER_PROVISION=1.

1. Install the package

Start with the published npm package

The operator flow starts here, then runs openclawbrain install from a normal shell.

npm install -g @openclawbrain/openclaw@0.3.0
2. Pick the attachment policy

Shared by machine, dedicated when you need isolation

A machine-level shared brain is acceptable if you want one memory surface. Use a dedicated activation root when you want stricter rollback, blame, or experimental separation. If multiple profiles point at the same root, add --shared.

openclawbrain install --openclaw-home "$OPENCLAW_HOME" --activation-root "$ACTIVATION_ROOT" --workspace-id "$PROFILE_NAME"
3. Verify the runtime truth

Check install, load, export, and serving separately

Restart when you want immediate runtime proof, then read the brain state from the same activation root.

env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway restart
openclawbrain status --activation-root "$ACTIVATION_ROOT"
openclawbrain daemon status --activation-root "$ACTIVATION_ROOT"
4. Operator truth is stronger

Status, daemon, and rollback give a better machine read

The operator surface can now distinguish hook install from load proof, seed state from promoted serving, and live passive-learning visibility from old snapshots.
