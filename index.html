Memory for OpenClaw

Evidence, not vibes, for agent memory.

OpenClawBrain helps OpenClaw agents understand that “close it out” means evidence, not vibes — and that the same correction should not have to be taught twice.

It is the difference between magic memory and an accountable local system: selective routing, SQLite search, bounded XML prompt injection, and proof surfaces you can inspect. The spine is simple: LLM decides semantic meaning. Code enforces trust boundaries. SQLite stores the graph and evidence.

Ollama proposes; code disposes. Local models propose structured JSON. Code validates, redacts, scopes, thresholds, and decides what gets stored or injected.

What it actually does

Remembers durable lessons

Corrections, preferences, workflows, and context become scoped memory nodes — not raw transcript dumps.

Keeps prompts small

Most memories stay silent. When a turn needs help, OpenClawBrain retrieves and injects only the small slice that matters now.

Shows its work

Status, doctor, proof, graph, search, route decisions, and injection records make memory inspectable instead of mystical.

One worked turn

You correct the agent: “Actually, use pnpm in this repo.”

OpenClawBrain distills that into a scoped correction. Later, when you say “Run the tests,” SQLite FTS retrieves the correction, ContextSelector ranks it, and the prompt receives a bounded block:

<openclawbrain_context>
Relevant memory:
- Must follow: Use pnpm in this repo.
</openclawbrain_context>

The agent runs pnpm test without you re-teaching the convention. Afterward, OpenClawBrain records whether the memory helped, failed, was ignored, or got corrected.

What is live today

OpenClawBrain is a real plugin today, not a concept page. The current release is v0.2.10. The core live loop has been verified end to end: temporary correction → SQLite memory → FTS retrieval → XML prompt injection → cleanup.

That does not mean every edge case is done. It is best described as production-dogfood beta: the core capture/store/retrieve/inject path works, while route-learning quality, status polish, and long-term organic runtime still deserve more miles.

Latency shape

In the dogfood snapshot used for the explainer, 157 of 158 turns — 99.4% — added zero synchronous model latency.

route decisions: 158
no_extra_llm: 157
enqueue_async_only: 1
sync_memory_planner: 0

Feedback distillation can take longer because it runs off the prompt path. A live local qwen2.5:32b-instruct verification took about 29 seconds without blocking the user turn.
