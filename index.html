OpenClawBrain: TypeScript-first brain packages for OpenClaw

OpenClaw owns runtime orchestration, fail-open behavior, and prompt assembly. OpenClawBrain owns contracts, event normalization, provenance, immutable pack artifacts, activation helpers, compiler output, and learner updates. The promoted pack is the only supported learning and serve boundary in this repo. This site describes both where the repo stands today and the target end shape it is building toward — explicitly and separately.

How it works

Most AI agents start every conversation from scratch. OpenClawBrain gives yours a memory that improves over time — a knowledge graph that stores structured relationships between facts, and a learned router (route_fn) that decides which facts to pull into the prompt for each query.

Step 1

Your agent runs on OpenClaw

OpenClaw handles sessions, tools, prompts, and answers — the runtime that actually talks to users. If anything goes wrong with the brain, the agent keeps working normally.

Step 2

OpenClawBrain builds a knowledge graph

Your agent's memory is a graph, not a pile of text chunks. Each node is a fact, a rule, or a correction. Edges capture how they relate. The learner materializes candidate packs off the hot path — immutable bundles of graph, vector, provenance, and router artifacts. Pack-level graph-dynamics fields (decay, co-firing, prune/merge counts) describe build-time evolution metadata, not live runtime mutation.

Step 3

A learned router picks what matters

At query time, the promoted pack's route_fn walks the graph and selects the right context. It blends graph structural priors (graph_prior) with per-query relevance (QTsim). Route updates are PG-only — trained from explicit labels off the hot path. A fresher candidate pack is materialized asynchronously and becomes serve-visible only after promotion by activation.

Current truth — proved in this repo today

What OpenClawBrain delivers now

  • Promoted-pack compilation: activation selects the active pack; compiler serves only from it; usedLearnedRouteFn, routerIdentity, checksum, and PG metadata prove which artifact ran.
  • PG-only route_fn: human, self, scanner, and teacher labels become explicit PG-ready training examples; no heuristic overlay in the serve path.
  • Off-hot-path candidate refresh: learner materializes fresher packs asynchronously; promotion makes them compile-visible.
  • Bootstrap attach from zero events: self-boots truthfully before first live export; seed state is operator-visible.
  • Operator observability: activation health, collected-label counts, PG objective metadata, route artifact diffs, rollback lineage, and explicit fallback usage.
  • Distribution today: local tarballs via pnpm release:check + pnpm fresh-env:smoke. Not npm-published yet.
  • Graph-dynamics: decay/co-firing/prune/merge counts are pack artifact metadata — not live runtime mutation of the active pack.
Target end shape — where the architecture is heading

What OpenClawBrain is building toward

  • Supported operator API: npm-published @openclawbrain/* wave gated by pnpm release:status showing a matching v0.1.2 tag.
  • Scanner/labels/harvest on by default after attach: all supervision inputs active from first promoted pack, not opt-in config.
  • Continuous graph update as live operations: decay, Hebbian co-firing, pruning, and reorganizing running on the active pack during serving — not just build-time metadata.
  • Full RL-native structural maintenance: connect, split, merge, and prune actions with conservative guardrails in the live serving runtime.
  • Hard API enforcement of PG-only constraint: package-level enforcement at every integration point, not just a documented convention.
  • Full OpenClaw / OpenClawBrain split at the API surface: operator-facing package API that expresses the runtime/learning boundary, not only docs.
  • Stronger proof ladder: shadow-mode and narrow online rungs reached with published artifacts; Brain Ground Zero benchmark family expanded.

Why graph-based routing outperforms RAG on benchmarks

RAG retrieves text chunks by similarity and hopes the right ones land in the prompt. That works for simple lookups but breaks down when facts depend on each other, queries are ambiguous, or accuracy compounds over time. OpenClawBrain replaces similarity search with learned selection over structure.

Difference 1

It learns from corrections

When a user corrects the agent, the correction is tied to the exact routing decision that produced the wrong answer. Next time a similar question comes in, the router picks differently. RAG has no feedback loop: correct it a hundred times, and it returns the same chunks tomorrow.

Difference 2

It follows relationships, not just similarity

RAG ranks isolated text chunks by how similar they sound to the query. The knowledge graph stores actual relationships — "this deploy policy depends on that rollback rule" — and the router follows those edges to reach context that matters even when the wording does not match.

Difference 3

It compounds over time

Every correction and outcome makes the routing function a little better. After a few hundred interactions the agent is measurably sharper — not because someone tuned it, but because the router learned what works. A static RAG index never improves.

Difference 4

It uses less context for better results

Because the router is selective, it sends fewer but more relevant context blocks to the model. In benchmarks, it uses roughly 5× less context per query than the best RAG baseline — while scoring 8–25 percentage points higher on accuracy. Less context also means faster, cheaper inference.

Two systems, one goal

OpenClaw and OpenClawBrain have distinct jobs. The split keeps serving fast and learning safe to iterate on.

OpenClaw owns runtime

Sessions, tools, prompt assembly, answers, and fail-open behavior. If the brain goes offline, your agent keeps working — it just serves without learned context until the next healthy pack is ready.

OpenClawBrain owns learning

Builds immutable pack artifacts (graph, vector, provenance, learned route_fn), stages candidates, promotes on activation, and exposes operator observability. Learned route updates are PG-only — no heuristic overlay. All artifact assembly runs off the hot path; the hot path always reads the promoted pack.

Quick start

Build the workspace and produce a brain pack in four commands. 

corepack enable
pnpm install --frozen-lockfile
pnpm check
pnpm release:check          # full proof lane + local tarballs
pnpm fresh-env:smoke        # clean-room outside-consumer install proof

Distribution today: local tarballs from .release/; not npm-published yet. Operator tools: pnpm operator:status, pnpm operator:doctor, pnpm operator:rollback -- --dry-run. Full setup guide: docs/setup-guide.md.
