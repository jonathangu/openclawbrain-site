Persistent brain for OpenClaw

Your OpenClaw agent should not have to relearn the same lesson twice.

OpenClaw starts each new session with a clean slate. OpenClawBrain gives it a persistent local brain that keeps corrections, pulls the right context for each ask, and gets sharper on work it sees again and again. The intended default is simple: one shared brain per machine is acceptable, dedicated roots stay available when you want tighter isolation, and OpenClaw stays fail-open if the learning side falls behind.

What changes in practice

OpenClaw still runs the agent. OpenClawBrain changes what the agent carries into the next job. In practice, that shows up in three places.

Fewer repeated mistakes

Corrections become reusable memory

When you fix a mistake, OpenClawBrain records what went wrong and why. When a similar situation shows up again, the agent can reach for that correction instead of stepping into the same hole.

Per-ask retrieval

The brain does not show the same memory every time

A trained routing function picks the context that matters for this ask instead of shoveling in every vaguely related chunk. Prompts get smaller, and two similar questions can still retrieve different memory when the task actually differs.

Better recurring workflows

Repeated workflows improve

Outcomes and feedback feed back into training. The workflows your agent runs every week start to tighten up because the router learns which context actually earns its keep.

How it works

Three product ideas, minus the jargon.

Knowledge graph

Facts live as connections, not loose documents

Instead of a pile of disconnected text chunks, knowledge lives in a graph. "This deploy policy depends on that rollback rule" becomes a real connection, not two files that merely happen to share a word.

Learned routing

A trained router decides what this ask needs

When a question arrives, the router walks the graph and chooses the nodes most likely to help this ask. It learns from corrections and outcomes, so over time it gets better at putting the right context on the table.

Structural plasticity

The memory graph reshapes as reality changes

Nodes connect, split, merge, and prune under policy control. Stale knowledge fades, live connections strengthen, and the ablations show this is the biggest single accuracy lift across every benchmark family.

Current status

Public technical alpha — stronger operator truth, same ambitious target

OpenClawBrain is still early alpha. The benchmark story is further along than the field-hardening story. The product direction is one shared brain per machine, with retrieval decided per ask and learning happening in the background, while dedicated roots remain available when you want tighter isolation. The public operator proof is now real: Eagle verified install, passive learner start, first export, promotion, learned route_fn, default bge-large, optional qwen3.5:9b teacher enablement, and later served route / graph truth updates, while Tern verified attach plus EXTERNAL RESTART on a Telegram-connected profile. That is narrow live operator proof, not broad production hardening or live answer-quality proof. Expect breaking changes before a stable release. Apache-2.0 licensed. Requires Node 20+.

Read the split literally. Benchmark proof says the memory architecture beats strong RAG baselines. Operator proof now says the install, passive-learning, and served-pack story is real on live machines. Broad same-gateway many-profile safety and shared-root concurrency guarantees are still separate proof work.

Install for OpenClaw operators

If you want to try it now, use the published package and run openclawbrain install from a normal shell. The current front door provisions bge-large by default, can enable a compatible local qwen3.5:9b teacher, and by default attaches your profile to a shared machine-level brain unless you deliberately choose a dedicated root.

Current honest read: Eagle proved install can auto-start the exact-root passive learner, reach first export, and serve from a promoted pack with live bge-large embeddings plus an enabled local qwen3.5:9b teacher. Tern proved attach plus EXTERNAL RESTART on a live Telegram-connected profile. A shared activation root per machine is acceptable when you want one memory surface, and different asks can still retrieve different context from that same brain. Use a dedicated root when you want rollback, disable, cleanup, or uninstall scoped to one profile. Skip embedder provision only with --skip-embedder-provision or OPENCLAWBRAIN_INSTALL_SKIP_EMBEDDER_PROVISION=1.

1. Install the package

Start with the published npm package

This is the front door for operators: install the package, then run openclawbrain install from a normal shell.

npm install -g @openclawbrain/openclaw@0.3.0
2. Attach a profile to the machine brain

Share by default, isolate on purpose

Reuse one activation root for the machine unless you want a profile to have its own rollback, disable, cleanup, or uninstall boundary. If multiple profiles point at the same root, add --shared.

openclawbrain install --openclaw-home "$OPENCLAW_HOME" --activation-root "$ACTIVATION_ROOT" --workspace-id "$PROFILE_NAME"
3. EXTERNAL RESTART and check-ins

Check install, load, export, and serving separately

Restart OpenClaw externally when you want immediate runtime proof, then use the same activation root for the operator check-in surfaces.

env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway restart
env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway status
openclawbrain status --activation-root "$ACTIVATION_ROOT"
openclawbrain daemon status --activation-root "$ACTIVATION_ROOT"
4. Operator truth and rollback

Scope operator actions deliberately

Status and daemon distinguish hook install from load proof, seed state from promoted serving, and live passive-learning visibility from old snapshots. Rollback, disable, cleanup, and uninstall follow the root you chose: dedicated roots scope those actions to one profile, while shared roots make them machine-wide.
