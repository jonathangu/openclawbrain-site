Install OpenClawBrain on an existing OpenClaw host

Install it, upgrade it, and verify it before you trust it.

OpenClawBrain turns the history you already have into a bounded memory layer for OpenClaw. It harvests conversations, memory notes, docs, and corrections, promotes only replay-passing packs, and serves a small live slice instead of stuffing raw history into every prompt.

Install first. Upgrade with the same command. Verify immediately.

The public operator story is intentionally compact. openclawbrain install is the install lane, the upgrade lane, and the repair lane for the selected OpenClaw home. The rest of the flow exists to prove that the runtime and the reporting surfaces agree.

New install

Attach OpenClawBrain to the host you already run

Use this when the host does not have OpenClawBrain yet. The command converges plugin packaging, repairs the hook if needed, and pins the activation root for the selected home. 

openclawbrain install --openclaw-home '~/.openclaw'
Upgrade / repair

Refresh an existing install without switching lanes

Use the exact same command after package-manager changes, host drift, or an awkward partial install. The public upgrade story is not a second workflow. 

openclawbrain install --openclaw-home '~/.openclaw'
Restart

Reload the gateway

Install converges the boundary. Restart makes the live runtime load the updated hook.

openclaw gateway restart
Verify

Compare detailed status

This is the fast surface for attachment, load proof, serving state, and learned route availability.

openclawbrain status --openclaw-home '~/.openclaw' --detailed
Proof

Capture the durable bundle only when you need it

The proof command packages summary, verdict, step logs, startup breadcrumbs, and runtime-load proof into one bundle directory.

openclawbrain proof --openclaw-home '~/.openclaw'
Activation and teacher wiring are separate. If you see BRAIN LOADED or the selected home shows as attached, the OpenClawBrain hook is wired correctly for that OpenClaw home. That does not by itself prove an optional teacher model is configured. Teacher wiring uses dedicated config fields such as brainTeacherEnabled, brainTeacherProvider, and brainTeacherModel, and the same status surface should report teacherConfigured=true, the expected provider/model, and teacherConfigError=null.
Path discipline matters: keep ~/.openclaw pinned on install, status, and proof. The restart command sits between them, but it does not take the path flag itself.

What happens to your history after install

OpenClawBrain does not dump your whole past into the prompt. It harvests the material you already surfaced, learns in the background, promotes only replay-ready packs, and serves bounded live context when it actually helps.

Authority

Corrections outrank stale recap

Explicit user corrections become durable current-truth memory. Summaries help navigation, but they do not win when they conflict with a direct correction.

Boundaries

Harvest is selective, not invasive

The harvester reads exported conversations, memory notes, docs, and corrections you already surfaced. It skips .env, binary junk, and unexported filesystem noise.

Live path

Bounded context stays on the hot path

The live runtime serves a small useful slice plus learned routing. If the brain has nothing useful or is unavailable, the agent still answers.

What it actually does

The product story should fit in one pass: harvest what matters, promote only proven structure, and serve bounded context instead of prompt sprawl.

Harvest

Read the useful parts of the past you already have

Exported turns, memory markdown, docs, and explicit corrections become candidate graph material. Summaries compact bulk history so the runtime can search it cheaply.

Promote

Learn in the background and only serve promoted packs

Candidate structure is not live by default. Replay gates promotion so the runtime serves proven packs rather than half-written hot-path mutations.

Serve bounded context

Inject a small useful slice, not your whole life history

The runtime serves the active promoted pack, a bounded structural budget, and the learned route_fn when available. That keeps latency and trust visible.

The shortest clean mental model is: learner = the background pipeline that watches events, builds candidate packs, and updates routing; teacher = the optional local model that adds supervision off the hot path; route_fn = the learned policy artifact the live runtime uses to pick which bounded context blocks to inject.

Why operators care

The point is not just “memory.” The point is operator reality: quicker installs, visible truth surfaces, less session amnesia, and a live path that stays bounded.

No raw-history prompt stuffing

Bulk history becomes searchable summaries and exact-source expansions when needed, not an unbounded prompt tax on every answer.

Learning stays off the hot path

Export, replay, and pack promotion happen outside the live answer step. The served runtime stays inspectable and predictable.

Fail-open behavior remains honest

If there is no useful memory match or the promoted pack is unavailable, the agent still runs instead of collapsing behind a broken memory dependency.
