Persistent memory for OpenClaw

Your OpenClaw agent should not have to relearn the same lesson twice.

OpenClaw starts each new session with a clean slate. OpenClawBrain gives it a memory that keeps corrections, pulls the right context for the job, and gets sharper on work it sees again and again. It stays local, sits behind OpenClaw, and fails open if the learning side falls behind.

What changes in practice

OpenClaw still runs the agent. OpenClawBrain changes what the agent carries into the next job. In practice, that shows up in three places.

Fewer repeated mistakes

Corrections become reusable memory

When you fix a mistake, OpenClawBrain records what went wrong and why. When a similar situation shows up again, the agent can reach for that correction instead of stepping into the same hole.

Cleaner prompts

Retrieval gets narrower

A trained routing function picks the context that matters for each question instead of shoveling in every vaguely related chunk. Prompts get smaller, and the useful signal stops drowning in filler.

Better recurring workflows

Repeated workflows improve

Outcomes and feedback feed back into training. The workflows your agent runs every week start to tighten up because the router learns which context actually earns its keep.

How it works

Three product ideas, minus the jargon.

Knowledge graph

Facts live as connections, not loose documents

Instead of a pile of disconnected text chunks, knowledge lives in a graph. "This deploy policy depends on that rollback rule" becomes a real connection, not two files that merely happen to share a word.

Learned routing

A trained router decides what to retrieve

When a question arrives, the router walks the graph and chooses the nodes most likely to help. It learns from corrections and outcomes, so over time it gets better at putting the right context on the table.

Structural plasticity

The memory graph reshapes as reality changes

Nodes connect, split, merge, and prune under policy control. Stale knowledge fades, live connections strengthen, and the ablations show this is the biggest single accuracy lift across every benchmark family.

Current status

Alpha — v0.1.11

OpenClawBrain is still early alpha. The benchmark story is further along than the field-hardening story. The routing function, graph structure, and plasticity policy are working; the published install, attach, and restart path is verified on real OpenClaw profiles; and live-learning proof on attached installs is still separate, with fresh brains sometimes sitting at awaiting_first_export until the first export lands. Expect breaking changes before a stable release. Apache-2.0 licensed. Requires Node 20+.

Install for OpenClaw operators

If you want to try it now, start with the published package and run openclawbrain install from a normal shell. The default path provisions bge-large, then attaches one dedicated brain to one OpenClaw profile.

The install, attach, and restart path is verified on real OpenClaw profiles. Keep one brain per profile. Skip embedder provision only with --skip-embedder-provision or OPENCLAWBRAIN_INSTALL_SKIP_EMBEDDER_PROVISION=1. The local teacher path is optional; if you turn it on through Ollama, the default model is qwen3.5:9b.

1. Install the package

Start with the published npm package

This is the front door for operators: install the package, then run openclawbrain install from a normal shell.

npm install -g @openclawbrain/openclaw@0.3.0
2. Attach one brain to one profile

Keep each OpenClaw profile isolated

Give each profile its own profile home and activation root so rollback and cleanup stay in their lane.

openclawbrain install --openclaw-home "$PROFILE_HOME" --activation-root "$ACTIVATION_ROOT" --workspace-id "$PROFILE_NAME"
3. Restart and verify

Use the clean OpenClaw restart shape

Restart the gateway, then check status with the same clean-shell pattern.

env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway restart
env -i HOME="$HOME" PATH="$PATH" openclaw --profile <PROFILE_NAME> gateway status
Rollback stays bounded

Detach without cross-profile spillover

One activation root per profile keeps rollback, disable, and cleanup fenced to that profile instead of splashing across the rest of the fleet.
