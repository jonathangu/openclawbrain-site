OpenClawBrain: Memory and Learned Routing for OpenClaw

OpenClawBrain adds durable memory and learned routing to OpenClaw, so repeated workflows improve over time instead of reloading the same context every turn.

  • Fast at runtime: bounded local retrieval on the hot path
  • Improves over time: background learning from traces and labels
  • Evidence-first: published benchmark artifacts with explicit claim boundaries

Who it's for

Teams running OpenClaw on repeated workflows

Use OpenClawBrain when you already run OpenClaw on repeated workflows — deploys, runbooks, triage, monitoring, research, or other recurring operator tasks — and you want the system to reuse what worked before instead of starting fresh every turn.

The problem

Plain retrieval answers the current query. It does not make the next similar query better.

OpenClawBrain adds a learned routing layer so successful paths get reinforced, bad paths get suppressed, and repeated workflows stop paying the full rediscovery cost every time.

How it works

Three steps, on and off the hot path

Step 1

Retrieve locally at runtime

Bounded local retrieval runs on the hot path. No full pre-scan gate before first useful routing.

Step 2

Capture signals from real use

Human labels, self labels, and scanner/harvester labels feed into one learner loop by default.

Step 3

Improve in the background

Async teacher updates and graph learning stay off the hot path. Routes that succeed get reinforced.

Evidence

What is proven today

Published recorded-session and sparse-feedback benchmark artifacts show OpenClawBrain outperforming baseline RAG systems under controlled conditions.

What is proven

Recorded-session H2H: full_brain 0.975 vs 0.896 (+7.9 pp). Sparse-feedback 10-seed: 91.96% vs 67.05% (+24.91 pp). Mechanism proof: learned 1.00 vs graph_prior_only 0.50.

What is not yet claimed

Live production answer-quality wins still require shadow-mode and narrow online artifacts. This claim is not made.
