Memory for OpenClaw

Your Claw should remember how you work.

OpenClawBrain is a local memory layer for OpenClaw. It learns from your corrections, preferences, workflows, and repeated feedback, then brings back the right context when it matters.

Not a giant transcript dump. Not a vague “AI memory” box. It is a small, inspectable system: a SQLite memory graph, local LLM updates, learned routing pointers, and bounded prompt injection.

Simple version: when you tell your Claw “don’t do that again,” OpenClawBrain is the part that helps it actually stop doing that again.

What it does

Remembers feedback

Corrections, preferences, project rules, and ways you like work done become scoped memories instead of disappearing after the turn.

Learns passively

It watches for useful signals in normal work: “use pnpm,” “keep this short,” “route Pelican work over there,” “closeout needs evidence.”

Injects carefully

Most memories stay out of the prompt. The code retrieves and injects only a small block when the current turn looks like it needs that memory.

The shape of the system

OpenClawBrain sits beside the normal OpenClaw run. It does not replace the main model. It gives the model better working memory.

your turn
 ↓
redact the packet
 ↓
route_fn decides whether memory should help
 ↓
SQLite FTS + graph search finds candidates
 ↓
ContextSelector chooses a small set
 ↓
inject bounded <openclawbrain_context>
 ↓
after the turn, feedback distillation updates the graph

The important bit is the split: prompt-time work stays small, while heavier learning can happen after the turn.

Why the graph matters

Flat memory gets messy fast. OpenClawBrain stores memories as nodes and links: corrections, preferences, workflows, context, route examples, outcomes, and superseded facts.

That lets the system answer better questions:

  • Is this a current rule or an old correction?
  • Does this apply to this agent, this repo, or everywhere?
  • Has this memory helped before, or did it cause mistakes?
  • Which related memories should travel together?

SQLite keeps it local and inspectable. FTS5 gives fast local search. The graph keeps memory from becoming one long note file.

How LLM updates fit in

The local LLM is used for judgment, not unchecked authority. It can propose that a turn contains a durable correction, a preference, a workflow, or a route-learning example.

Then code decides what happens next: redact, validate JSON, check scope, dedupe, threshold, store, reject, or mark for later. That is the core trust boundary.

LLM decides semantic meaning. Code enforces trust boundaries. SQLite stores the graph and evidence.

Learned route_fn pointers

Over time, OpenClawBrain learns which kinds of turns usually need memory. A test command may need repo tooling rules. A writing request may need your style preference. A Pelican task may need routing guidance.

Those learned pointers help route_fn decide quickly: search memory, stay silent, or spend one bounded planner call if the turn is ambiguous. The goal is simple: remember the useful thing without slowing every turn down.

Install in a minute

openclaw plugins install clawhub:openclawbrain@0.2.16
openclaw plugins enable openclawbrain
openclaw gateway restart

That is the short path. The install page has the config, Ollama path, multi-agent setup, and checks.
