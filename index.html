A learned memory layer for OpenClaw agents

Your agent gets a second brain — one that learns

Most AI agents start every conversation with amnesia. They search their notes by keyword and hope for the best. OpenClawBrain is different: it gives your agent a trail guide instead of a search bar. The guide builds a map of everything your agent knows — facts, rules, corrections, how they connect — and learns which knowledge to pull in for each question. Wrong answer? The guide remembers the mistake and routes around it next time.

How it works

Think of it like hiring a librarian for your agent. The librarian doesn't just search by keyword — they know the collection, understand how topics connect, and learn from every time you say "no, not that one."

Step 1

Your agent runs on OpenClaw as usual

OpenClaw handles the live conversation — sessions, tools, prompts, answers. It's the front desk. If the brain is ever unavailable, the agent keeps working normally. No single point of failure.

Step 2

OpenClawBrain builds a connected map of knowledge

Instead of a pile of text snippets, the brain organizes knowledge as a graph — a map where each node is a fact, rule, or correction, and the connections show how they relate. "This deploy policy depends on that rollback rule" is a connection, not just two documents that happen to use similar words.

Step 3

A trained guide picks the right knowledge

When a question comes in, the brain's routing function walks the map and selects the most useful context — not just the most similar-sounding text. It blends structural knowledge (what's connected to what) with per-query relevance. The guide improves over time: corrections and outcomes feed back into training, so the same mistake doesn't happen twice.

Works today — you can verify it yourself

What you get right now

  • Versioned brain packs: The brain's knowledge ships as immutable snapshots. You can see exactly which version is active, roll back to a previous one, and audit what changed.
  • A routing function that learns from real feedback: Human corrections, automated checks, and outcome signals all train the router. No hand-tuned rules in the serving path.
  • Background learning, zero disruption: New brain versions are built in the background. They go live only when you promote them — no surprise changes mid-conversation.
  • Cold start that works: The brain bootstraps honestly from zero data. You can see its seed state before it's ever been trained.
  • Operator dashboard: Health checks, label counts, routing diffs, rollback history — you can see what the brain is doing at every step.
  • Local distribution: Install via local tarballs today (pnpm release:check). npm packages coming soon.
Building toward — not claiming yet

The destination

  • npm-published packages: @openclawbrain/* on npm, with proper release gating.
  • Learning on by default: Feedback collection active from the moment you attach — no extra config needed.
  • Live graph evolution: The knowledge map reshapes itself in real time — forgetting stale facts, strengthening frequently-used connections — not just at build time.
  • Hard API boundaries: The learning-only constraint enforced in code at every integration point, not just documentation.
  • Stronger proof ladder: Shadow-mode testing, narrow live-traffic evidence, and expanded benchmarks.

Integration guide

Why this beats the usual approach (RAG)

RAG (retrieval-augmented generation) is the standard way to give an AI agent knowledge: chop documents into chunks, search by similarity, stuff the closest matches into the prompt. It's a keyword search in a costume. OpenClawBrain is more like a guide who has walked the terrain and keeps learning it.

Difference 1

Corrections actually stick

Tell a RAG system it gave you the wrong answer, and it forgets by tomorrow — it'll return the same chunks next time. With OpenClawBrain, each correction is tied to the routing decision that caused the mistake. The guide reroutes next time.

Difference 2

It follows connections, not just keywords

RAG finds text that sounds similar to your question. The brain's map stores real relationships — "this deploy policy depends on that rollback rule" — so it reaches relevant knowledge even when the wording doesn't match.

Difference 3

It gets sharper over time

Every correction, every outcome makes the guide a little better. After a few hundred interactions, the agent is measurably more accurate — not because someone hand-tuned it, but because the routing function learned from experience. A static RAG index never improves.

Difference 4

Less context, better answers

Because the guide is selective, it sends fewer but more relevant pieces of knowledge to the model. In benchmarks, it uses roughly 5× less context per query than the best RAG baseline — while scoring 8–25 percentage points higher on accuracy. Less context also means faster, cheaper responses.

Two systems, one goal

Think of it as front-of-house and back-of-house. The split means your agent stays fast and reliable, while learning happens safely in the background.

OpenClaw = the front desk

Handles live conversations: sessions, tools, prompt assembly, answers. If the brain goes offline, the agent keeps working — it just serves without learned context until the next healthy brain version is ready.

OpenClawBrain = the back office

Builds versioned brain packs (the knowledge map, the trained routing function, provenance records), stages new candidates, and promotes them when ready. Learning never touches the live path — it runs in the background, and the live agent always reads from the last promoted version.

Quick start

Four commands to build the workspace and produce your first brain pack. 

corepack enable
pnpm install --frozen-lockfile
pnpm check
pnpm release:check          # runs all proof tests + builds local packages
pnpm fresh-env:smoke        # verifies a clean install works end to end

Distributed as local tarballs for now (npm packages coming). Operator tools: pnpm operator:status, pnpm operator:doctor, pnpm operator:rollback -- --dry-run. Full setup: docs/setup-guide.md.
