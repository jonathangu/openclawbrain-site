What is OpenClawBrain?

Your AI agent gets smarter with every conversation

OpenClawBrain makes OpenClaw stateful and self-improving. It learns what context your agent should see for a given query — and gets better at it continuously from real use, feedback, and history.

How it works

Most AI agents start every conversation from scratch. OpenClawBrain gives your agent a memory that improves over time. The core idea: a knowledge graph plus a learned router that decides which memories to surface for each query — and gets sharper from every interaction.

Step 1

Your agent runs on OpenClaw

OpenClaw handles sessions, tools, prompts, and answers. It is the runtime — the thing that actually talks to users.

Step 2

OpenClawBrain builds a knowledge graph

It organizes your agent's memory as a graph of context blocks — facts, rules, corrections, and relationships — not a flat list of text chunks. The graph captures structure: which pieces of knowledge relate to each other and how.

Step 3

The learned router picks what matters

At query time, a learned routing function walks the graph and selects the right context to serve. It combines durable structure (what the graph already knows) with query-specific signals (what fits this particular question). Over time, it gets better — surfacing more relevant information with less noise.

Why this is better than ordinary RAG

Standard RAG (retrieval-augmented generation) stores text chunks and retrieves the nearest neighbors by semantic similarity. It works for simple lookups, but it has fundamental limits. OpenClawBrain is different in four ways that matter.

Difference 1

It learns from corrections

When a user corrects the agent, the router learns from it. Next time a similar question comes up, the routing is better. Ordinary RAG has no feedback loop — it returns the same results no matter how many times you correct it.

Difference 2

It serves relationships, not just similar text

The knowledge graph stores structured relationships between pieces of context. The router can follow edges — "this deploy rule connects to that rollback policy" — instead of just matching keywords. RAG only sees text similarity.

Difference 3

It gets sharper over time

The routing function improves continuously from real usage, feedback, and corrections. After a few hundred interactions, it surfaces better context with less noise. A static RAG index stays exactly the same from day one.

Difference 4

It uses less context for better results

Because the router is selective, it sends less but more relevant context to the model. In benchmarks, it uses roughly 5× less context per query than the best RAG baseline — while scoring significantly higher on accuracy.

Two systems, one goal

OpenClaw and OpenClawBrain have distinct jobs. This split is intentional — it keeps the runtime fast and the learning safe.

OpenClaw owns runtime

Sessions, tools, prompt assembly, answers, fail-open behavior. If something goes wrong with learning, your agent keeps working.

OpenClawBrain owns learning

Watches usage and feedback. Builds brain packs. Trains the route function that decides what context to surface. All of this happens off the hot path — it never slows down responses.

Evidence

Two benchmark families are published and reproducible. They compare OpenClawBrain's learned routing against standard RAG baselines on frozen workloads — same queries, same data, same scoring rules.

Recorded-session benchmark

800 queries, 8 baselines. Learned routing scores 97.5% accuracy vs best RAG baseline at 89.6%. That means the learned router gets the right answer on about 1 in 13 queries where RAG gets it wrong.

Sparse-feedback benchmark

Even with limited feedback signals, learned routing hits 92% vs RAG at 67% — while using roughly 5× less context per query. The router is both more accurate and more efficient.
