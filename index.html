Canonical operator lane

One home. One restart. One status check. One proof bundle.

Install or repair

openclawbrain install --openclaw-home '~/.openclaw'

Reload the gateway

openclaw gateway restart

Verify detailed status

openclawbrain status --openclaw-home '~/.openclaw' --detailed

Capture proof when you need it

openclawbrain proof --openclaw-home '~/.openclaw'

Keep the same selected home pinned on every openclawbrain command. The restart command sits between install and verify, but it does not take the path flag itself.

Residual noise

Fresh sterile proof can still warn

Truth ladder: if a fresh host has healthy detailed status but proof still looks noisy, trust status --detailed first and treat proof as durable evidence, not as the only health check.

Install first. Restart, verify, and prove only when needed.

The public operator lane is intentionally short: install, restart, status, proof. That is enough to get a new host live and enough to repair a healthy one without changing the boundary.

Install

Make the host attachable

The install command converges the plugin boundary and pins the selected home for OpenClawBrain.

Restart

Reload the live runtime

Restart is the handoff between install and verify. It makes the gateway load the updated hook.

Status / proof

Check truth, then capture evidence

Detailed status is the fast compare surface. Proof is the durable bundle when scrollback is not enough.

Split packages

Runtime and operator concerns stay separate

@openclawbrain/openclaw carries the runtime payload; @openclawbrain/cli owns the install/status/proof lane.

Upgrade is the same lane, rerun for an existing host.

There is not a separate magic second workflow today. When an existing host needs repair, refresh, or package reconciliation, rerun the install lane on the same pinned home and then verify.
