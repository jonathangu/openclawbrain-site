OpenClawBrain

Your OpenClaw agent should not relearn the same lesson twice.

OpenClawBrain is a learned retrieval brain for OpenClaw. It's a fork of lossless-claw that adds a paper-faithful reinforcement learning layer for context routing.

It combines lossless transcript memory (never forget the conversation) with a knowledge graph whose routing policy learns from outcomes (get better at what context to surface).

Featured paper: Jonathan Gu's 2016 reinforcement-learning paper. It lays out the corrected policy-gradient update that informs the project's learning direction.

Start with the GitHub repo — it is a good front page for the project and has the README, SPEC, and architecture notes.

The problem

AI coding agents repeat the same mistakes. You correct them, they forget. You teach them a pattern, it's gone next session. You figure out that gh pr create works better than hub, but the agent keeps suggesting hub.

Context management tools like lossless-claw solve the memory problem — they never lose what happened. But they don't solve the learning problem — knowing what context to surface for a given task.

OpenClawBrain solves both.

Two layers, one plugin

Layer 1: Lossless transcript memory

Every message persisted in SQLite. Older chunks summarized into a DAG. Nothing ever lost. Agents can grep, describe, and expand any past conversation.

Inherited from lossless-claw. Battle-tested.

Layer 2: Learned retrieval brain

A knowledge graph with corrections, toolcards, workflows, and episode anchors. Routing policy learns from real outcomes via reinforcement learning.

New. Paper-faithful. Tested (302 tests).

Sacred rule: the summary DAG is not the brain graph. They are complementary systems connected by bridge nodes.

How it works

Every turn, the brain traverses a knowledge graph to find the best context for this specific query:

  1. Seed — find start nodes by embedding similarity to the query
  2. Expand — follow outgoing edges to candidate nodes
  3. Route — compute softmax policy over candidates + STOP
  4. Fire — add chosen nodes to context (respecting budget)
  5. Repeat until STOP chosen, budget exhausted, or max hops reached

The assembled context has four sections, by priority:

  1. Correction cards — human-taught fixes, always surface first
  2. Route-selected evidence — nodes the learned policy picked
  3. Toolcards and playbooks — when/how to use specific tools
  4. Transcript support — fresh tail + archival recall from the summary DAG

How it learns

After each interaction, the brain harvests reward signals from four sources:

SourceTrustExamples
HumanHighest"No, that's wrong", "Perfect!", brain_teach corrections
SelfHighTool success/failure, test pass/fail, deploy result
ScannerMediumRepeated patterns, workflow structure, co-firing nodes
TeacherLowOff-path LLM evaluation (sees only what the router saw)

These signals update edge weights via the paper's REINFORCE algorithm. Positive outcomes strengthen the routing path that led to them. Negative outcomes weaken it. Over time, the brain converges on routing policies that surface the right context.

Structural graph mutations

The brain doesn't just learn weights — it restructures itself:

MutationWhen
SplitA node has mixed content and only part fires
MergeNear-duplicate nodes always co-fire
ConnectSuccessful episodes repeatedly bridge two regions
PruneEdge weight decayed near zero, no recent traversal
InjectHuman correction or novel successful pattern

All mutations go through a replay gate before promotion. The live serving graph is never mutated during queries.

The paper

The routing policy update implements Lemma 6.1 from:

Jonathan Gu — Reinforcement Learning (Econometrics Field, July 2016)

∂/∂ρ vρ(st) = E[ z · Σl=tT ∂log Pρ(al|sl) / ∂ρ ]

The key insight: Williams (1992) REINFORCE only looks one step ahead — "what is the best single move?" The corrected update sums over the entire trajectory from step t to terminal state T — "given everything that follows, what should I have done?"

This assigns credit to every routing decision in the episode, not just the last one. A good deployment outcome strengthens the entire path that led to the right context being surfaced.

Paper assumptions honored in the implementation

  • Assumption 1: Finite-time game — traversal bounded by maxHops (default: 8)
  • Assumption 2: Reward only at terminal state — no intermediate rewards during traversal
  • Stochastic policy: Softmax over edges + STOP — never argmax, even at low temperature
  • Monte Carlo rollouts: Each traversal is a complete episode from seed to STOP

Get started

Install

openclaw plugins install @jonathangu/openclawbrain

Or link a local checkout: openclaw plugins install --link /path/to/openclawbrain

That's it

The brain initializes automatically on first load. It discovers your workspace sources (markdown, code, configs, OpenClaw memory files), chunks them structure-aware, computes embeddings, and builds a seed graph.

Teach it

Use brain_teach in any conversation to add corrections, toolcards, or workflows. Use brain_status to check health. Use brain_trace to see exactly why specific context was selected.
