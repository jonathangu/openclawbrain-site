OpenClaw-native memory

Start using the brain now. Let it keep learning in the background.

OpenClawBrain is the long-lived memory and routing layer for OpenClaw. It plugs into ongoing agent work immediately, keeps query-time routing local and bounded, and continuously trains a learned runtime route_fn from historical replay, scanner/harvester labels, async teacher labels, and user feedback.

Tight OpenClaw integration OpenClaw asks the brain for a bounded [BRAIN_CONTEXT] block before response generation, then feeds corrections back through the same session flow.
Fast startup Build from a workspace, start the daemon, and query right away. Historical replay and maintenance continue later without blocking first use.
Real learned routing Ultimate Policy Gradient updates the route policy from labels. The teacher stays off the hot path; the learned router runs on it.

What OpenClawBrain actually does

This is not a generic memory pitch. It is a concrete operating shape for OpenClaw agents.

1. Build once from the workspace

Index the OpenClaw workspace into a portable state.json. That gives you an immediately usable brain instead of waiting on a full training run.

2. Serve a bounded memory path

Run openclawbrain serve so OpenClaw can query the brain on each relevant turn and append a deterministic [BRAIN_CONTEXT] block.

3. Keep training in the background

Replay historical sessions, harvest ongoing sessions, and add corrections and teachings without pausing the agent. Training follows the work, not the other way around.

4. Ship a learned route policy

The product value is the learned runtime route_fn: it uses the current query to choose better graph edges on the hot path, based on labels accumulated offline.

Hot path and shadow path

The separation is the product. Fast retrieval stays local; expensive labeling and updates stay asynchronous.

Runtime path

What runs during a live OpenClaw turn

  1. OpenClaw packages the inbound turn as a stable summary plus chat_id.
  2. The brain embeds the query and seeds candidate memory nodes.
  3. The learned route_fn scores possible next hops and emits a bounded route.
  4. OpenClaw appends the resulting [BRAIN_CONTEXT] block to the prompt.
  5. Feedback stays keyed to the same chat_id for later learning.
Learning path

What trains the router between turns

  1. The scanner/harvester layer turns historical and ongoing sessions into labels and route examples.
  2. The async teacher replays difficult or ambiguous decisions and proposes better targets.
  3. Ultimate Policy Gradient unifies human feedback, self-learning, harvester signals, and async-teacher labels in one update rule.
  4. The next deployment window serves the updated route policy, not the teacher.

Quick start
python3 -m pip install openclawbrain

WORKSPACE=~/.openclaw/workspace
BRAIN_DIR=~/.openclawbrain/main

mkdir -p "$BRAIN_DIR"
openclawbrain init --workspace "$WORKSPACE" --output "$BRAIN_DIR"
openclawbrain serve --state "$BRAIN_DIR/state.json"

python3 -m openclawbrain.openclaw_adapter.query_brain \
  "$BRAIN_DIR/state.json" \
  "what matters for deploy approvals?" \
  --chat-id smoke-test \
  --format prompt \
  --exclude-bootstrap

That gets the brain into the loop immediately. After that, start replaying sessions and scheduling maintenance.
