OpenClawBrain

Your agent should not relearn the same lesson twice.

OpenClaw gives your agent a runtime. OpenClawBrain gives it memory. It keeps corrections, brings back the right context for this task, learns from whether the last attempt worked, and lets you inspect why a choice was made.

Get Started in 30 Seconds

openclaw plugins install @jonathangu/openclawbrain
openclawbrain init /path/to/your/workspace

Why do you need this?

You teach the agent something once. It forgets. You tell it to keep answers concise, use bullets, and end with the next action, and before long you are teaching the same lesson again. That is not intelligence. That is expensive forgetfulness.

Corrections can win immediately

Just correct the agent in plain language, and that lesson can become durable memory instead of a one-session comment that evaporates.

Context becomes task-specific

The goal is not to dump every vaguely related note into the prompt. The goal is to bring back the few pieces most likely to help this task right now.

Repeated workflows get sharper

When one path leads to a good outcome and another leads to a bad one, the routing policy learns from that difference over time.

It shows its work

Commands like openclawbrain trace and openclawbrain doctor let you inspect what surfaced and why.

How it works

Think of OpenClawBrain as a five-step teaching loop.

1. Remember prior conversations and corrections

Past sessions are not treated like disposable chat logs. Useful corrections and recurring workflows can become durable memory.

2. Find the part of memory that fits this task

When a new ask arrives, the system searches for the neighborhood of past knowledge that is actually relevant, instead of treating every memory as equally important.

3. Surface only the best pieces at prompt time

The goal is not maximum recall at any cost. The goal is to hand the agent the few pieces of context most likely to help right now.

4. Learn from outcomes over time

If one retrieval path helps produce a good outcome, it grows stronger. If another path wastes time or leads to failure, it loses weight.

5. Trace why the system chose what it chose

This is not supposed to be magical. You can inspect the route, the surfaced memory, and the decision trail behind an answer.

What a real correction looks like

A real interaction is simple: the user says something, the agent gets it wrong, the user corrects it, and the next answer gets better.

User says

How should you answer me?

Agent says

I’ll answer in long freeform paragraphs.

User says

Wrong — keep it concise, use bullets, and end with the next action.

Agent says

Got it. I’ll keep answers concise, use bullets, and end with the next action.

User says

How should you answer me?

Agent says

Concisely, with bullets, and ending with the next action.

What got retrieved behind the scenes

  • the user's correction is committed as durable memory
  • on the next similar question, that correction is retrieved as priority context before the agent answers
  • the stale default answer loses because the retrieved correction outranks it
  • brain_trace shows the retrieved context, the fired correction, and the losing path
