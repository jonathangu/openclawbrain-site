How it works

Small memory, at the right time.

OpenClawBrain is local, accountable memory for OpenClaw agents. It remembers durable corrections, preferences, workflows, and context, then retrieves only the small slice that matters for the current turn.

LLM decides semantic meaning. Code enforces trust boundaries. SQLite stores the graph and evidence.

The loop

user turn
 ↓
route: should memory help?
 ↓
search: SQLite FTS + memory graph
 ↓
select: choose a tiny relevant slice
 ↓
inject: bounded XML context
 ↓
learn: update the graph after the turn

Retrieval stays small. Learning can happen later.

Why a graph

OpenClawBrain stores memories as scoped nodes and links: corrections, preferences, workflows, context, route examples, outcomes, and superseded facts.

That helps it answer practical questions: does this apply here, is it still current, has it helped before, and which related memories should travel together?

Why the LLM boundary matters

The LLM is used for semantic judgment. It can notice that a messy correction is actually durable feedback.

But the LLM does not write directly to memory. Code redacts, validates, scopes, dedupes, thresholds, and stores only what passes.

Learned routing pointers

Over time, OpenClawBrain learns which kinds of turns usually need memory. A test command may need repo tooling rules. A writing task may need style preferences. A project task may need routing guidance.

The goal is not to search memory every time. The goal is to remember when it helps.
