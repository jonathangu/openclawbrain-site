The system in one pass

OpenClawBrain does not do one thing. It runs a loop: store the past, turn it into a graph, learn better routes in the background, promote what survives replay, then serve only the small slice that helps now.

Step 1

Store the raw material

It keeps transcripts, corrections, documents, tool traces, status evidence, and operator notes as durable memory instead of leaving them trapped in old chats.

Step 2

Turn memory into a graph

Nodes hold facts, corrections, examples, traces, and artifacts. Edges start simple, then become useful: override, evidence, related step, good next hop.

Step 3

Learn better routes off the hot path

Replay, labels, and the async teacher score traversal choices in the background. Candidate packs improve there, not in the live user request.

Step 4

Serve a bounded useful slice

The runtime injects the few pieces that matter before the same mistake repeats. It does not dump the whole past into every prompt.

The graph is the point

This is not just a bigger transcript store. OpenClawBrain organizes memory into a graph so retrieval can move from “what happened near what” to “what helps answer what.”

Nodes

What gets stored

Corrections, examples, tool results, docs, status facts, summaries, and replay evidence can all become nodes.

Edges

How memory gets structure

Edges start as time and task adjacency. Later they can mean override, support, belongs-with, or likely-next-useful-step.

Packs

What the runtime actually serves

The live system serves promoted packs, not half-written state. That keeps rollback simple and the hot path predictable.

How graph traversal works

Query-time traversal must stay fast. So the runtime does not ask a live LLM what to do at every hop. It uses a small learned route function to decide which branches to expand, which memories to pull forward, and when to stop.

Runtime routeFn

Fast local policy

The runtime route function is the hot-path policy. It scores edges against the current query and decides what is worth expanding.

Branching

More than top-k search

Traversal can fan out across multiple edges. It can stop one branch early and keep exploring another when that is better for the final answer.

STOP matters

Knowing when not to keep digging

A good memory system learns when more retrieval will help and when it will only add latency, noise, or token cost.

Important: the runtime path should feel intelligent without paying the price of per-hop live LLM calls. That is why the learned routeFn is central.

Where labels come from

OpenClawBrain does not trust one source of feedback. It combines several, with a clear hierarchy.

  • Human feedback is strongest. If a user corrects the agent, that should outweigh stale recap material.
  • Self-learning matters when the system can see that a route helped or hurt a later outcome.
  • Harvester signals capture durable structure from the artifacts themselves.
  • Async teacher labels provide dense weak supervision at many traversal decision points.

In practice, the teacher provides most of the volume. Human correction provides the highest trust.

The async teacher and dreaming loop

The teacher is not there to slow down the live runtime. It runs in the background. That is where the dreaming loop lives.

Replay

Revisit real past queries

Replay asks which paths were useful, which branches were distractions, and which stops were right.

Teacher

Label decision points

An async teacher model can label route choices densely without blocking the live request path.

Dreaming

Keep improving off to the side

Dreaming is the passive background loop: generate more labels, update the route policy, and apply improvements when state locks allow.

Promotion is the safety rail

Learning does not write straight into the live runtime. Candidate packs have to survive replay and health checks first.

Candidate

Build new memory off-path

New graph state and route updates land in candidate packs first.

Replay-gated

Check whether the route is actually better

Replay and health checks decide whether a candidate route helps, hurts, or should stay out of the live path.

Promoted

Serve only what passed

The runtime serves the promoted pack. That is what makes rollback and operator proof possible.

Worked example: one Teacher v3 lifecycle

A skeptical reader should be able to trace one path from proposal to outcome. This is the public example shape: propose, replay, guard with bounded canary discipline, then promote or stay shadow-only.

1. Proposal Evidence attached

A compiler or lint proposal carries lineage, evidence, and counterevidence so the next step is inspectable.

2. Replay Real traces

The proposal is replayed against real candidate state so the decision comes from behavior, not wishful prose.

3. Canary Bounded and off by default

Canary discipline keeps the runtime path explicit: it only opens when replay and proof say the change is ready.

4. Outcome Promoted or shadow-only

If the proposal wins, it is promoted. If it loses or stays unready, it remains shadow-only and target-state stays labeled as such.

Shipped vs target-state: this lifecycle is the shipped path. Broader class coverage and any future rollout shapes are still target-state until they land.

What makes OCB special

The trick is not “more memory.” The trick is learned bounded retrieval.

  • It uses a graph, not just a flat transcript dump.
  • It learns a fast local route function instead of depending on live per-hop LLM calls.
  • It keeps teaching and labeling off the hot path.
  • It promotes and serves packs instead of mutating live memory directly.
  • It can fail open. If memory is unavailable, the agent still runs.
