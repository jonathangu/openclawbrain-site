How it works

A memory graph that stays out of the way.

OpenClawBrain learns from ordinary feedback, stores the lesson locally, and injects it only when the current turn needs it.

The product idea is simple. You should not have to teach your Claw the same thing twice. The code is built around that idea: route first, search locally, inject a small block, then learn from the outcome.

LLM decides semantic meaning. Code enforces trust boundaries. SQLite stores the graph and evidence.

The short version

OpenClawBrain is a native OpenClaw plugin. It hooks into the prompt build, looks at the current turn, and asks a practical question: would memory help here?

If yes, it searches local SQLite memory, ranks the candidates, and adds a compact <openclawbrain_context> block. If not, it stays quiet.

After the turn, it looks for durable feedback and updates the graph in the background.

A worked example

You say:

Actually, use pnpm in this repo.

OpenClawBrain treats that as a likely correction. It redacts the turn, asks the local LLM for structured JSON, then code validates the proposal before anything is stored.

{
  "type": "correction",
  "scope": "repo",
  "content": "Use pnpm instead of npm in this repo",
  "confidence": 0.94,
  "tags": ["tests", "package-manager"]
}

Later you say:

Run the tests.

route_fn recognizes a test/build turn, pulls the package-manager correction through SQLite FTS, and the selector injects a small reminder:

<openclawbrain_context>
Relevant memory:
- Must follow: Use pnpm instead of npm in this repo.
</openclawbrain_context>

The agent runs pnpm test. Afterward, OpenClawBrain records whether that memory helped, was ignored, or got corrected again.

The main code path

before_prompt_build
  → redact current turn
  → route-fn.ts makes a fast memory plan
  → latency-controller.ts decides if a planner call is allowed
  → memory-store.ts searches SQLite FTS
  → context-selector.ts ranks and formats memories
  → inject bounded prompt context

agent_end / after_tool_call
  → feedback-distiller.ts proposes memory updates
  → memory-operations.ts validates and applies them
  → learning.ts updates outcome scores
  → route-learning.ts updates route examples and policy snapshots

That separation is deliberate. Retrieval has to be fast and boring. Learning can be more thoughtful because it runs after the prompt path.

What gets stored

OpenClawBrain stores distilled operational memory, not raw chat logs.

MemoryWhat it meansExample
correctionYou corrected behavior that should change next time.Use pnpm in this repo.
preferenceA stable way you like work or writing done.Short, direct, natural writing.
workflowA reusable way to handle a task.Run build, test, and proof before closeout.
contextBackground that changes future decisions.Route Pelican work to the Pelican workspace.
route exampleA learned pointer for when memory should participate.Writing requests should check style memory.

Each memory can carry scope, confidence, importance, freshness, tags, links, evidence hashes, and outcome counters. That is what turns memory from a notes file into a graph.

How the graph helps

The graph keeps related lessons together without dumping everything into the model.

  • A correction can supersede an older correction.
  • A repo rule can stay scoped to one repo.
  • A writing preference can travel with writing tasks, not build tasks.
  • A bad memory can decay or get pruned after poor outcomes.

Search starts with SQLite FTS, then graph expansion can bring in connected memories when they are useful. The selector still has the final budget, so the prompt stays small.

LLM updates, with a hard boundary

The LLM is good at semantic judgment. It can see that “make this sound like a person wrote it” is probably a writing preference, not a random complaint.

But the LLM does not write to memory directly. It returns structured JSON. Code checks the JSON, redacts content, applies scope, dedupes against existing memories, rejects weak claims, and writes to SQLite only if the proposal passes.

Default local path: Ollama through a localhost OpenAI-compatible endpoint. If the local model is down, the main agent keeps working and memory gets quieter.

Learned route_fn pointers

route_fn is the front door. It decides whether memory should participate before the system spends budget.

At first, it uses local rules and cached policy. Over time, route learning adds examples: this kind of turn usually needs repo memory; that kind usually needs writing style; this one should stay silent.

turn shape: "help me write a post"
learned pointer: check writing-style memories
budget: small
sync LLM call: usually no

That is why the system can learn without turning every prompt into an extra planning call.

Latency shape

TierWhat happensBlocks the turn?
0Local route decision only.No.
1Cached route + SQLite retrieval.No model call.
2One bounded local planner call for ambiguous/high-signal turns.Yes, capped.
3Distillation, route learning, pruning, policy updates.No. Background.

The goal is not “memory at any cost.” The goal is useful memory with predictable latency.
