OpenClawBrain lifecycle

What actually happens when you install the brain.

OpenClawBrain adds a learned memory layer to OpenClaw without moving learning work onto the live response path. The runtime serves promoted packs, not partially written state, so useful context stays bounded and latency stays predictable.

This page is the end-to-end guide: first install on an existing OpenClaw host, what replay means, what learning means, what healthy status looks like, and what changes over time after install.

Start with the architecture

OpenClawBrain has two public pieces. @openclawbrain/openclaw is the plugin and runtime package inside OpenClaw. @openclawbrain/cli is the operator CLI that installs, verifies, captures proof, and manages the host-side lifecycle.

Runtime / plugin layer

During prompt build, the installed extension reads the current promoted pack, compiles a bounded slice of context, and injects it into the prompt. If the memory layer is unavailable, the agent still runs.

Operator / learning layer

After the response path finishes, OpenClawBrain exports turns and evidence, builds candidate packs, runs replay and health checks, and promotes a new pack only if it passes.

The key boundary: live serving stays fast and bounded. Learning happens later, off the response path.

What install does on an existing OpenClaw host

Install does not train a giant model on the spot. It converges one selected OpenClaw home onto the supported runtime layout, repairs the hook if it drifted, and pins the activation root that the runtime serves from.

Canonical operator lane

openclawbrain install --openclaw-home ~/.openclaw
openclaw gateway restart
openclawbrain status --openclaw-home ~/.openclaw --detailed
  • install targets one OpenClaw home, writes or repairs the hook for that home, and pins the activation root the runtime serves from.
  • openclaw gateway restart reloads the runtime so the installed hook can actually load into the live agent process.
  • status --detailed verifies the selected home and reports whether the runtime is attached, loadable, and proven on the exercised host surface.

Healthy signals include loadProof=status_probe_ready and attachTruth ... runtime=proven. Once a promoted pack exists, healthy detailed status should also report serveState=serving_active_pack and, where surfaced, routeFn available=yes. Startup breadcrumbs can also show BRAIN LOADED after the gateway restart.

First-run truth: immediately after install there may be no promoted pack yet. That is normal. The runtime can be attached before learned serving is active.

What replay means

Replay is the gate between a candidate pack and the live runtime. Instead of trusting a freshly built pack immediately, OpenClawBrain replays the relevant evidence and checks whether the candidate reproduces the right retrieval and reporting behavior.

Why replay exists

It keeps the live path on promoted packs instead of half-written state. A candidate that cannot survive replay does not get served.

What replay protects

It protects installation truth, runtime truth, and learning truth from drifting apart. The runtime should only serve a pack that passed the promotion gate.

In practice, replay is part of the candidate-pack build and promotion flow: export turns, build a candidate pack, run replay and health checks, and promote only if that candidate passes.

What learning means

Learning in OpenClawBrain is not a single one-time training job. It is the background process that turns real usage into better promoted packs over time.

1

Export

After the agent replies, OpenClawBrain exports the turns and evidence from that completed response path.

2

Build

The learner turns those exports into a candidate pack instead of editing the live runtime state in place.

3

Replay and check

Replay and health checks verify whether the candidate deserves promotion.

4

Promote

Only a passing candidate becomes the new active promoted pack for live retrieval.

Why this matters: live answers stay predictable while the system keeps learning from real work in the background.

What changes over time

The brain becomes more useful gradually. Install prepares the host. Real turns create exports. Exports become candidate packs. Passing candidates become promoted packs. Promoted packs are what the live runtime actually serves.

  • Right after install: the host can be attached and healthy even if no learned pack is serving yet.
  • After real usage: export and learning have material to process.
  • After first promotion: detailed status can truthfully report serveState=serving_active_pack.
  • Over longer usage: better routes and useful retrievals can become repeatable behavior instead of one-off prompt luck.

If the memory layer is missing, broken, or not yet ready, OpenClawBrain fails open and the agent still replies normally.

How to verify and capture durable proof

Start with detailed status for the fast operator check. When you need one durable bundle with the exact commands, logs, and verdict, capture proof as a separate follow-up command.

Fast verify

openclawbrain status --openclaw-home ~/.openclaw --detailed

Durable proof

openclawbrain proof --openclaw-home ~/.openclaw

Today, proof remains a separate follow-up surface. It captures the install / restart / status / runtime evidence bundle with summary and raw artifacts.
