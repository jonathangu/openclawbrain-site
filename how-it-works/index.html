First principle: what do we do with the history and memory markdown files?

The answer should be boringly honest: we do not dump every old markdown file into every prompt. We keep operator-readable history surfaces, compact them into navigable memory structures, and let explicit current-truth corrections outrank stale recap when the two disagree.

Human-facing source of truth

Durable files such as memory notes, summaries, and operator-edited markdown stay legible and auditable. They are the surface a human can inspect, edit, diff, and reason about.

Runtime-facing source of truth

The runtime turns long history into a searchable abstraction: LCM summaries help the system decide where to look, raw expansion gets exact detail, and typed correction memory decides what should currently win when history conflicts.

Why this split matters

Summaries are navigation, not authority. They are cheap enough to search broadly, but they should not override an explicit user correction or a more exact raw quote.

What the promoted pack serves

The live prompt path serves a bounded slice from the active promoted pack. That means old markdown and old transcript history influence routing and recall, but they do not become an unbounded prompt tax.

One-line rule: use summaries as a search abstraction over history; use typed correction memory as the durable current-truth layer.

What the graph looks like after INIT

Right after init, the graph is a scaffold, not a miracle. That is the correct product behavior. Init should make the runtime attachable and inspectable, not pretend the system already learned from months of real work.

Current host initial pack

The current host still retains an initial pack with 8 blocks, 0 human labels, and 0 structural ops for split / merge / prune / connect. That is the pre-learning baseline the system grows from.

The honest story is: install gives you a live attachment boundary, not a mature memory city. The graph becomes interesting only after real turns, explicit corrections, trace exports, replay, and promotion.

Then what happens: raw history, candidate graph, promoted graph

The graph evolves in layers. First you have raw usage. Then the learner forms candidate structure from exports, labels, and traces. Then replay decides whether that candidate deserves promotion into the live runtime.

  • Raw usage: real turns, explicit corrections, and traces start as ordinary substrate history.
  • Candidate graph: exports and learning create a provisional structure that is still not live.
  • Promoted graph: only replay-passing structure becomes the runtime-visible active pack.

What replay actually does

Replay is the gate between “we learned something” and “the live runtime is allowed to believe it.” Without replay, the system would be one bad candidate away from mutating its serving truth on the hot path.

How to visualize this nicely: the deep detailed plan

The right visualization system should not be “a random force-directed graph because graphs are cool.” It should tell a sequence of truths at increasing depth: operator truth, lifecycle truth, graph-shape truth, and replay-decision truth.

Layer 1 — operator truth

Status-first cards

Start with a date-stamped host snapshot: proof verdict, promoted pack, route function state, graph size, traced routes, and structural ops. This is the “is it real?” layer.

Layer 2 — lifecycle story

Timeline before graph hairballs

Show init → raw turns → candidate pack → replay gate → promoted serving. Most people understand the system once that sequence is visually explicit.

Layer 3 — graph anatomy

Progressive graph reveal

Start with tiny post-init structure, then show which blocks become stronger over time, which nodes were pruned, which merges occurred, and which new connections were added.

Layer 4 — replay microscope

Why this pack won

Let operators inspect a candidate pack, the replay checks it passed or failed, and the exact reasons promotion happened or did not happen.

Layer 5 — current-truth overlay

Authority vs recap

Make it visually obvious which items are summary navigation, which items are raw-source expansions, and which items are explicit correction memories that currently win.

Layer 6 — interactive operator mode

Time scrubber + diff mode

The ideal final surface is a scrubber that lets you compare after init, after first export, after replay, and current promoted state with node/edge diffs.

Design rule: the first view should answer “is this real?”; the second should answer “what changed?”; the third should answer “why does the runtime trust this?”

Operator commands for reproducing the story yourself

If you want to reproduce the whole lifecycle on your own host, do it in this order: converge install, restart the runtime, inspect detailed status, then capture a durable proof bundle. 

openclawbrain install --openclaw-home ~/.openclaw
openclaw gateway restart
openclawbrain status --openclaw-home ~/.openclaw --detailed
openclawbrain proof --openclaw-home ~/.openclaw
  • install repairs or writes the hook, pins the activation root, and converges the host onto the supported layout.
  • gateway restart reloads the live runtime so the hook can actually load.
  • status --detailed shows whether you are still in init-only territory or already serving a promoted pack.
  • proof captures the durable bundle you can audit later.
