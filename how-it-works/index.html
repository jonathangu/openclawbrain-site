The system in one pass

OpenClawBrain follows a simple loop: remember useful past work, learn from it in the background, then bring back only the small slice that helps now.

Step 1

Store the raw material

It keeps useful past work, corrections, documents, and operating notes as durable memory instead of leaving them trapped in old chats.

Step 2

Organize memory into something useful

Instead of treating old chats like one long blob of text, OpenClawBrain organizes them so the system can find the part that actually helps.

Step 3

Learn in the background

It compares what helped and what did not in the background, so live user requests do not turn into slow online training experiments.

Step 4

Bring back only the few things that matter

When the agent needs help, OpenClawBrain brings back a small useful slice of memory. It does not dump your whole history into every prompt.

This is more than search

This is not just a bigger transcript store. OpenClawBrain organizes memory so it can move from “what happened near what” to “what is actually useful right now.”

What gets stored

Related pieces get connected so the system can move from one useful clue to the next instead of relying on one blind text match.

What the live system actually uses

The live system uses checked memory snapshots, not half-written state. That keeps rollbacks simpler and live behavior more predictable.

How it chooses what to remember

This choice has to stay fast. OpenClawBrain does not ask a live model to rethink the whole problem at every step. It uses a small learned chooser to decide which memories to bring forward and when to stop.

Fast chooser

Pick what is worth bringing back

The runtime quickly scores which memories look useful for the current request and ignores the rest.

Flexible search

Follow more than one path when needed

If one memory thread is not enough, OpenClawBrain can branch out and follow a few related ones instead of forcing everything through one top match.

Restraint matters

Know when to stop

A good memory system knows when more retrieval will help and when it will only add noise, delay, and token cost.

Important: the live path should feel smart without getting slow or bloated. That is why OpenClawBrain uses a fast learned chooser instead of dragging a full model into every retrieval step.

How it learns what helped

OpenClawBrain does not trust one source of feedback. It combines several, with a clear pecking order.

  • Your corrections matter most. If you tell the agent something was wrong, that should beat stale recap material.
  • Later outcomes matter too. If one memory path clearly helped or hurt, the system should learn from that.
  • Artifacts and documents contribute structure from the work itself.
  • Background review models can add a lot of extra labeling without blocking the live request path.

In practice, background review provides most of the volume, but direct human correction carries the highest trust.

Learning happens in the background

The learning loop is there to improve the system without slowing down live requests. That work happens in the background.

Replay

Revisit real past queries

Replay checks real past work and asks what actually helped, what distracted the agent, and when the system should have stopped sooner.

Background review

Score the choices

Background models can score lots of memory choices without getting in the way of live user requests.

Background updates

Keep improving off to the side

The system keeps refining what it remembers and how it chooses memory, but it does that off to the side, not in the middle of your live chat.

Nothing changes live without checks

Learning does not write straight into the live system. New memory snapshots have to pass checks first.

Draft

Build new memory off to the side

New memory ideas land in draft snapshots first.

Checked

Make sure it is actually better

Replay and health checks decide whether a draft helps, hurts, or should stay out of the live path.

Live

Serve only what passed

The live runtime only serves the checked snapshot. That is what makes rollback and proof possible.

How a change earns its way in

A skeptical reader should be able to trace one path from idea to outcome. The shape is simple: suggest a change, test it on real past work, open the door carefully if it looks good, then either keep it live or keep it out.

1. Suggestion Why this change?

A change should come with clear reasons and evidence, not just wishful thinking.

2. Replay Real traces

The change gets replayed against real past work so the decision comes from behavior, not just a nice story.

3. Careful rollout Small and explicit

If the change looks good, it should still open carefully and explicitly, not all at once.

4. Outcome Live or held back

If the change wins, it can go live. If it does not, it stays out of the live path.

Important: the point is not to let every idea change the live system. The point is to let good changes earn their way in.

What makes OCB special

The trick is not “more memory.” The trick is bringing back the right small piece of memory at the right time.

  • It organizes memory instead of just storing a giant transcript dump.
  • It learns a fast local way to choose memory instead of doing slow full-model retrieval at every step.
  • It keeps learning work in the background instead of slowing down live chats.
  • It checks changes before they go live instead of mutating live memory directly.
  • It can fail open. If memory is unavailable, the agent still runs.
