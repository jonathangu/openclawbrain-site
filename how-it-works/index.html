How it works

OpenClawBrain: How It Works

Evidence, not vibes, for agent memory.

Imagine an agent that has worked with you for months — one that understands “close it out” means evidence, not vibes. OpenClawBrain makes those lessons available at the moment they matter, without turning every prompt into a giant diary.

This is the difference between magic memory and an accountable local system.

LLM decides semantic meaning. Code enforces trust boundaries. SQLite stores the graph and evidence.
Ollama proposes; code disposes.

TL;DR

OpenClawBrain is a local memory layer for OpenClaw agents: it learns durable corrections, preferences, workflows, and context, then retrieves only the small slice that matters right now. It uses selective routing, SQLite search, bounded prompt injection, and outcome learning instead of stuffing history into every prompt.

In v0.2.11, retrieval and capture are separate decisions. Retrieval stays conservative, while aggressive capture evaluates durable instructions, corrections, workflows, routing rules, outcomes, and explicit recall rules. The installed live loop is verified end to end: turn → capture audit → strict distillation/storage → SQLite/FTS retrieval → bounded prompt injection.

What it is not

It is not a vector database that blindly injects nearest-neighbor chunks. It is not raw transcript storage. It is not an always-on extra model call before every turn. It is not a remote SaaS memory service by default. And it is not a replacement for the main agent model.

It is a local reliability layer around the normal OpenClaw run: route, retrieve, inject a bounded memory block when useful, then learn whether that memory helped.

One annotated turn

Suppose you corrected the agent earlier:

Actually, use pnpm in this repo.

OpenClawBrain distills that into a scoped correction node with type, content, scope, confidence, importance, freshness, tags, evidence hash, and outcome counters.

Later, you say:

Run the tests.
  1. OpenClaw emits before_prompt_build.
  2. OpenClawBrain redacts the turn and RouteFn recognizes a build/test request.
  3. SQLite FTS retrieves the pnpm correction.
  4. ContextSelector ranks it highly because it is both relevant and corrective.
  5. The agent prompt receives a compact XML block.
<openclawbrain_context>
Relevant memory:
- Must follow: Use pnpm in this repo.
</openclawbrain_context>

The main agent runs pnpm test instead of npm test. After the turn, OpenClawBrain records whether the memory helped, was ignored, or led to another correction. Memory is not just retrieved; it is graded.

Retrieval mechanics

Retrieval is a funnel, not a dump.

current turn
 ↓
RouteFn: should memory participate?
 ↓
retrieval plan: queries, memory types, graph depth, budget
 ↓
SQLite FTS search
 ↓
graph expansion for connected memories
 ↓
ContextSelector ranking
 ↓
max-items / max-chars budget
 ↓
inject a small block, or stay silent

Corrections become “Must follow.” Preferences become “User preference.” Workflows become “Workflow.” Superseded, deleted, low-confidence, and over-budget memories stay out of the prompt. The important product behavior is restraint: most stored memories never touch a given turn.

Storage model

OpenClawBrain stores distilled operational memory, not chat logs.

TypeMeaningExample
correctionUser corrected previous behavior“Use pnpm in OpenClawBrain.”
preferenceStable user/project preference“Keep Telegram replies concise.”
workflowReusable way to do work“For Bountiful Garden, coordinate through @bbountiful_bot.”
contextBackground that changes future decisions“Route Pelican work to the Pelican workspace unless Jon explicitly redirects main.”

SQLite is the durable store. FTS5 is the local text index. The graph is explicit enough to inspect, prune, supersede, and learn from.

Feedback distillation runs off the prompt path. The live local qwen2.5:32b-instruct E2E check took about 29 seconds, stored one temporary memory, injected it, and cleaned it up afterward.

Ollama and the trust boundary

The default route/planner/feedback/learning model is qwen2.5:32b-instruct, reached through the local OpenAI-compatible endpoint http://127.0.0.1:11434/v1/chat/completions.

The model is used for semantic judgment: whether an ambiguous turn should retrieve memory, which candidates are relevant, whether a correction is durable enough to store, and whether an injected memory helped or caused harm.

The model does not write directly to memory. It proposes JSON. Code validates, redacts, scopes, deduplicates, thresholds, and writes. Ollama proposes; code disposes.

Degraded path: Ollama is down

If the local model is unavailable, RouteFn still runs locally, SQLite FTS still retrieves known memories, bounded prompt injection can still work, and background learning can retry later. The main agent keeps working; memory gets quieter instead of becoming a single point of failure.

The operator should be able to answer whether the plugin is enabled, SQLite is healthy, which latency tiers are being used, which memories were injected recently, and whether those injections helped or got corrected. That is the difference between magic memory and an accountable local system.
