OpenClaw Runtime Integration

March 2026 · v12.2.6+ series

This post describes the shipped boundary between the two systems. OpenClaw owns the live session: orchestration, prompt assembly, runtime diagnostics, and fail-open behavior. OpenClawBrain owns the learning side: normalized contracts/events, brain packs (versioned, immutable bundles of context and routing weights), activation (switching which pack is live, or rolling back to a previous one), deterministic compilation (the compiler always produces the same output from the same pack), and background learning.

Current runtime flow

OpenClaw with OpenClawBrain packages

User message
  ↓
OpenClaw runtime boundary normalizes events + workspace context
  ↓
Learned route function scores candidates from existing files + active pack state
  ↓
Deterministic compile assembles a bounded compiled context block for prompt assembly
  ↓
Model answers with stronger task-relevant prior
  ↓
Background learner harvests labels and writes candidate pack updates asynchronously

Why this changes outcomes

  • Fast startup: context starts from existing files and prior exports.
  • Fewer turns: learned routing sends task-relevant priors into prompt assembly before generation.
  • Teacher stays off the live request path: heavy learning work runs asynchronously while the learned routing function serves requests inline. The user never waits on the teacher model.
  • Instant rollback: brain packs are immutable once built. You can switch which pack is active or roll back to a previous one without any data-shape rewrites.

Runtime ownership boundary

OpenClaw runtime owns: channels, sessions, orchestration, prompt assembly, runtime diagnostics, and fail-open control.

OpenClawBrain owns: memory structure, event/export normalization, workspace metadata + provenance, brain pack assembly (packs are immutable once built), activation (promoting a new pack or rolling back to a previous one), deterministic compilation (same pack always produces the same output), and background learning. The relevant packages: @openclawbrain/contracts, @openclawbrain/events, @openclawbrain/event-export, @openclawbrain/workspace-metadata, @openclawbrain/provenance, @openclawbrain/pack-format, @openclawbrain/activation, @openclawbrain/compiler, and @openclawbrain/learner.

Flow, budgets, and safety defaults

Safe defaults:

  • Keep retrieval context bounded and measurable so comparisons stay fair.
  • Use bounded data-only compiled context blocks, not raw tool traces.
  • Run fail-open by default: if retrieval fails, continue normal OpenClaw flow.

Deep path:

  • Prioritize new incoming events first while historical ingestion and graph updates continue in background loops.
  • Serve traffic from the active pack immediately; learner output lands as candidate packs for explicit promotion.

Keep human labels, self-label harvesting, and scanner ingestion default-on; that is how routing quality improves continuously after boot.

Security checklist (optional but recommended)

  1. Block sensitive paths before event/export normalization runs.
  2. Apply secret redaction before retrieval payloads are assembled.
  3. Keep prompt input data-only and cap retrieval context aggressively.
  4. Audit provenance traces for every context payload used on the live request path.

How to enable (package-first)

corepack enable
pnpm install
pnpm check
pnpm release:pack

Then wire the released OpenClawBrain packages into the OpenClaw runtime boundary; fail-open behavior remains enabled as the production safety default.

How this interacts with learning and harvester

  • Corrections, human labels, and self-harvested labels are part of the default loop, not optional add-ons.
  • Learner output is written as candidate packs; activation handles switching to the new pack (or rolling back) without changing runtime boundaries.
  • Background learning generates candidate packs with updated routing weights. Structural graph fields (decay, co-firing, pruning) are build-time pack metadata today; live graph plasticity during serving is the target end shape, not the current reality.
  • Routing quality improves because retrieval and labels are tied to real session flow; each promoted pack carries better weights.

Proof boundary (keep claims honest)

  • Mechanism proof: bounded-context routing and learning mechanics improve route quality under controlled tests.
  • Product proof: end-to-end operator outcomes require full evaluations (turns, quality, latency, spend) and rollout evidence.

Rollback and OpenClaw compatibility

Rollback is immediate: switch to the previous active brain pack (or disable runtime-boundary wiring entirely) and restart the gateway. Because packs are immutable and the compiler is deterministic, OpenClaw serving behavior remains available without any data-shape rewrites.
