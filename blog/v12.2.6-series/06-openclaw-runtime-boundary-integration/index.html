How the guide plugs into OpenClaw

March 2026 · v12.2.6+ series

Think of OpenClaw and OpenClawBrain like a restaurant: OpenClaw is the front of house — it takes orders, assembles plates, handles service, and manages the dining room. OpenClawBrain is the kitchen — it prepares ingredients, learns which recipes work, and packages everything so the front can serve it fast. Neither side reaches into the other's domain.

What happens when a question comes in

The flow, step by step

User asks a question
  ↓
OpenClaw normalizes the request (captures context about files and workspace)
  ↓
The learned guide scores candidates: which pieces of knowledge matter for this question?
  ↓
The compiler assembles a bounded context block — only the relevant facts, no raw dumps
  ↓
The model answers with a stronger starting point
  ↓
In the background, the learner harvests feedback and builds better knowledge snapshots

Why this changes outcomes

  • Fast startup: context starts from existing files and prior exports — no cold-start wait.
  • Fewer back-and-forth turns: the guide sends task-relevant knowledge into the prompt before the model generates, like a colleague who briefs you before a meeting instead of making you ask 10 questions.
  • Learning never slows a live response: heavy learning work runs asynchronously in the background. The guide serves requests inline using what it already knows. You never wait on the learning process.
  • Instant rollback: knowledge snapshots are frozen once built. Switching to a previous snapshot is like swapping a book on a shelf — no data migration, no rewriting, just point to the old one.

Who owns what

OpenClaw (front of house): channels, sessions, orchestration, prompt assembly, diagnostics, and fail-safe control. If anything goes wrong, OpenClaw continues normally without the guide.

OpenClawBrain (kitchen): knowledge structure, event normalization, workspace metadata and audit trails, knowledge snapshots (frozen once built), activation (promoting a new snapshot or rolling back), deterministic compilation (same snapshot always produces the same output), and background learning. The packages: @openclawbrain/contracts, @openclawbrain/events, @openclawbrain/event-export, @openclawbrain/workspace-metadata, @openclawbrain/provenance, @openclawbrain/pack-format, @openclawbrain/activation, @openclawbrain/compiler, and @openclawbrain/learner.

Safety defaults and bounded context

Safe defaults:

  • Context is bounded and measurable, so comparisons stay fair and prompts stay manageable.
  • The model sees clean, structured context blocks — not raw tool traces or dumps.
  • Fail-open by default: if the guide can't find relevant context, OpenClaw continues its normal flow. No silent failures.

Under the hood:

  • New incoming events are prioritized; historical ingestion and knowledge map updates continue in background loops.
  • Traffic is served from the active snapshot immediately; the learner's output lands as candidate snapshots for explicit promotion — nothing changes until an operator says so.

Human corrections, automated labels, and self-harvested signals are on by default. That's how the guide's quality improves continuously after setup — like a librarian who gets better every week because patrons naturally tell them when something's off.

Security checklist (recommended)

  1. Block sensitive paths before event normalization runs.
  2. Apply secret redaction before context payloads are assembled.
  3. Keep prompt input data-only and cap context size aggressively.
  4. Audit the provenance trail for every context payload on the live path.

How to enable

corepack enable
pnpm install
pnpm check
pnpm release:pack

Then wire the released packages into the OpenClaw runtime; fail-safe behavior stays on as the production default.

How learning fits in

  • Corrections, human labels, and automated signals are part of the default loop, not optional add-ons.
  • The learner writes candidate snapshots; activation handles switching to the new one (or rolling back) without changing anything about how OpenClaw serves.
  • Background learning generates snapshots with improved routing weights. The knowledge map's structure is set at build time today; a map that reshapes itself during live serving is the target, not the current reality.
  • Quality improves because the guide's learning is tied to real session flow — each promoted snapshot carries better knowledge about which facts matter for which questions.

Where the evidence stands

  • Mechanism proof: the guide and learning machinery work correctly under controlled tests.
  • Product proof: end-to-end outcomes on real traffic require full evaluations (turns, quality, latency, cost) and live rollout evidence. That's still ahead.

Rollback and compatibility

Rollback is immediate: switch to the previous active snapshot (or disable the integration entirely) and restart. Because snapshots are frozen and the compiler is deterministic, OpenClaw's serving behavior is always available without migration or data rewrites. It's like pulling a book off the shelf and putting the old one back — the library keeps running.

Enablement details and operational defaults are documented in the canonical integration guide.
