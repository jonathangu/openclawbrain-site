The learned runtime route function

March 2026 | v12.2.6+ series

The real product value is not "shadow routing" as a concept — it is the learned routing function (route_fn) that OpenClaw calls on every live query. This is the function that decides which knowledge to pull into the prompt, and it is the thing that gets better over time.

What the route function does

At query time, the brain needs to decide which edges to follow from the currently active memory nodes in the knowledge graph. It blends two sources of information: what the graph already knows about how facts relate to each other, and what the current query specifically needs.

The two sources:

  • graph_prior — the graph's durable structural knowledge, accumulated from earlier learning and graph maintenance. This encodes which facts connect, which corrections supersede older guidance, and which context blocks tend to fire together. It is what makes the brain useful even before it has seen the current query.
  • QTsim — a query-specific relevance score that asks "which next hop fits this particular turn?" This is closer to what RAG does, but applied selectively within the graph rather than over a flat chunk index.
  • A learned confidence gate decides how much to trust each source for each routing decision — not a fixed blend, but an adaptive one.

Confidence mixing

The router should not trust the query-specific signal equally on every decision. Sometimes the query gives a strong, unambiguous signal — "I need the deploy freeze policy" — and the router should lean hard on QTsim. Other times the query is vague — "what should I know about this?" — and the graph's structural knowledge is more reliable. The confidence gate uses uncertainty features (entropy and margin of the QTsim scores) to shift weight automatically:

lambda = sigmoid(a0 + a1 * margin - a2 * entropy)
route_fn = lambda * QTsim + (1 - lambda) * graph_prior

How the runtime route function is trained

The live router is learned from labels produced outside the live request path — none of this work blocks the user:

  • scanner / harvester labels from historical and ongoing sessions
  • async teacher labels on difficult or ambiguous route choices
  • human corrections and outcome feedback from live OpenClaw use

The Ultimate Policy Gradient (a single learning rule that combines all these feedback sources, with human corrections taking priority) is what lets those labels update one runtime policy instead of turning into disconnected heuristics.

Why the teacher stays out of the live request path

The teacher model exists to produce training labels, not to answer the current turn. If it sat in the live path, the system would just be adding another inference dependency — slower and more expensive, with no learning benefit on the current query. The whole point is that the learned route_fn handles live queries cheaply and fast, while the teacher runs in the background to make the next version of that router better. This is the same separation that makes the system safe to deploy: if background learning fails or is delayed, the live router keeps working with its current weights.
