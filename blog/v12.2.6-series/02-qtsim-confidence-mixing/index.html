Post 2

The learned runtime route function

March 2026 | v12.2.6+ series

The phrase that matters is not just "shadow routing." The product value is the learned runtime route_fn that OpenClaw calls during a live query.

What the route function does

At query time, the brain needs to decide which edges to follow from the currently seeded memory nodes. The runtime policy combines durable graph structure with query-conditioned evidence.

In the current framing:

  • graph_prior carries durable structural preference from earlier learning.
  • QTsim is the query-conditioned component that asks which next hop fits this turn.
  • The learned gate decides how much to trust each one for this decision.

Confidence mixing

The route function should not trust the query-conditioned signal equally on every decision. The gate uses uncertainty features like entropy and margin to shift weight between the durable prior and the query-conditioned score.

lambda = sigmoid(a0 + a1 * margin - a2 * entropy)
route_fn = lambda * QTsim + (1 - lambda) * graph_prior

How the runtime route function is trained

The live router is learned from labels produced off the hot path:

  • scanner / harvester labels from historical and ongoing sessions
  • async teacher labels on difficult or ambiguous route choices
  • human corrections and outcome feedback from live OpenClaw use

Ultimate Policy Gradient is what lets those labels update one runtime policy instead of turning into disconnected heuristics.

Why the teacher still stays off the hot path

The teacher is there to produce labels, not to answer the current turn. If the teacher sits in the live request path, the system is no longer selling a learned runtime router. It is just selling another inference dependency.
