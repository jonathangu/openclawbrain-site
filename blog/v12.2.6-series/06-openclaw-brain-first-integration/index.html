Post 6: Brain-first OpenClaw Integration

March 2026 · v12.2.6+ series

Brain-first mode is the recommended OpenClaw integration pattern. It is optional, and it is safe to run as a gradual rollout. Enabling it does not replace OpenClaw behavior — it adds one additional preference-aware context retrieval step before model generation and keeps learning wired through the same correction loop.

What exactly changes when enabled?

Before brain-first (OOTB OpenClaw)

User message
  ↓
OpenClaw loads AGENTS/role + local files
  ↓
Model answers directly from loaded context
  ↓
Response sent
  ↓
Manual teach/correction workflow (if you built one)

After brain-first

User message
  ↓
OpenClaw calls adapter hook (query_brain) with chat_id
  ↓
Brain returns [BRAIN_CONTEXT] candidates (cap-limited)
  ↓
Hook appends [BRAIN_CONTEXT v1]...[/BRAIN_CONTEXT] to prompt
  ↓
Model answers with stronger task-relevant prior
  ↓
Same-turn corrections -> capture_feedback -> correction node + context learning

Why this changes outcomes

  • Fewer turns: model gets structured memory context before asking for clarification.
  • Less manual context engineering: repeated file-and-history reconstruction is replaced by learned routing signals.
  • Cleaner correction loop: `chat_id` lets correction updates target recent fired route candidates, not generic free-form memory logs.

Flow, budgets, and safety defaults

Safe defaults:

  • --max-prompt-context-chars 12000 for query context (default recommendation for OpenClaw hooks).
  • Use the [BRAIN_CONTEXT v1] delimiter, not raw JSON or extra metadata.
  • Start with bootstrap exclusions enabled so duplicated AGENTS/SOUL/USER/MEMORY material is not re-sent.

Deep path:

  • For tool-output ingestion (e.g., media transcripts) keep --tool-result-max-chars 20000 as a practical cap.

Security checklist (optional but recommended)

  1. Install the hook through a shell wrapper instead of embedding logic directly in AGENTS.md.
  2. In the wrapper, block sensitive paths from ever reaching retrieval context.
  3. Apply redaction for keyword matches before any query payload leaves the wrapper.
  4. Keep output as data-only brain context and do not pass full tool traces into user prompt.
export OPENCLAWBRAIN_EXCLUDE_PATHS=".env,.openclawbrain,dist,build"
export OPENCLAWBRAIN_REDACT_KEYWORDS="api_key,secret,token,password,private_key"
export OPENCLAWBRAIN_MAX_PROMPT_CONTEXT_CHARS=12000

OpenClaw hook behavior should remain fail-open by default: if brain lookup fails, continue on with the normal OpenClaw flow. That preserves availability while still benefiting from the memory path whenever it works.

How this interacts with learning and harvester

  • The learning model is unchanged: corrections still resolve by chat_id through capture_feedback and replay paths.
  • Harvester + maintenance schedules keep running exactly as before.
  • Brain data becomes higher quality faster because retrieval and correction are linked to actual session flow instead of manual reminders.

Why this does not break OpenClaw OOTB

Brain-first is opt-in; OpenClaw defaults remain untouched. If you remove the hook lines from AGENTS.md or disable their execution flag, the agent returns to existing OpenClaw flow immediately. No schema, config, or storage migration is required to turn it off.

Enablement details and operational defaults are documented in the canonical integration guide: docs/openclaw-integration.md.
