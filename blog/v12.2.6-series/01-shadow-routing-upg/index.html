Post 1

Shadow routing + Ultimate Policy Gradient

March 2026 | v12.2.6+ series

OpenClawBrain is easiest to understand if you separate two paths. The live path answers queries right now — it runs inside OpenClaw and must be fast. The learning path runs in the background, digesting corrections and feedback to produce better routing weights for future turns. The two paths never block each other.

Shadow routing

"Shadow routing" means the expensive teacher model never sits in the live request path — the path the user is waiting on. When a query arrives, OpenClaw calls the learned routing function (route_fn), which walks the knowledge graph and picks bounded context for the prompt. That is the entire live-path cost. Replay, scanner/harvester passes, and the background teacher all run later — on historical and new data — to produce updated routing weights that take effect on future turns.

live request path:  OpenClaw -> activate brain pack + compile -> learned route function -> bounded context for prompt
background path:    sessions -> scanner/harvester labels + background teacher labels -> Ultimate Policy Gradient -> updated route function

Where the labels come from

The update rule draws from multiple label sources, because OpenClawBrain is built to live inside a real OpenClaw workflow.

  1. Human feedback: corrections and teachings tied to the same turn_id as the fired route.
  2. Outcome signals: positive or negative outcomes on turns the brain actually helped serve.
  3. Scanner / harvester labels: structured signals mined from historical and ongoing session data.
  4. Async teacher labels: background teacher judgments on ambiguous or difficult route decisions.

Ultimate Policy Gradient (UPG)

The Ultimate Policy Gradient is a single learning rule that folds every feedback source — human corrections, outcome signals, scanner/harvester labels, and teacher judgments — into one routing policy. There are no separate policies for each source. They merge with a clear priority order: human corrections carry the most weight, then direct outcomes, then automated signals. A single correction from a user can override thousands of weaker automated labels — exactly the behavior you want in a product that learns from real people.

Human corrections should be able to override automated labels. That is a product requirement, not just a training detail — if a person tells the system it routed wrong, that signal wins.

Why this matters for OpenClaw

  • You can use the brain immediately after init and serve — no waiting for background learning to finish. The graph's built-in structure already provides useful routing on day one.
  • Replay and teacher labeling happen asynchronously — they never block first use or add latency.
  • The thing that improves over time is the learned route_fn that OpenClaw actually calls on every turn — not a separate model, but the same routing function, with better weights.
