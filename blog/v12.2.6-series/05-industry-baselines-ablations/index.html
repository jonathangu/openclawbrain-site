Post 5

Baselines and the comparison contract

March 2026 | v12.2.6+ series

OpenClawBrain should compare itself against the retrieval systems operators would actually use instead of hiding behind vague "AI memory" framing. It also needs a stricter contract for when those comparisons are strong enough to publish.

Baseline set

BaselineWhy it belongsHarness mode
Vector top-k Cold-start floor when there is no learned routing yet. vector_only
Deterministic rerank / legacy edge similarity Represents classic retrieval-plus-rerank stacks without a learned route policy. edge_sim_legacy
Graph prior only Shows what happens when structure exists but the query-conditioned route component is absent. graph_prior_only
QTsim only Shows what happens when the query-conditioned component acts without the durable structural prior. qtsim_only
Learned mixed router The actual OpenClawBrain runtime route policy. learned

What the current site can say honestly

  • The mechanism can be reproduced locally with deterministic simulations and ablations.
  • The harness can compare routing modes on recorded query sets.
  • The site does not yet publish broad head-to-head operator wins from real OpenClaw traffic.

What stronger comparison evidence looks like

StageUseStill too weak for broad product claims?
Simulation + ablation Mechanism proof Yes
Offline recorded-session eval Head-to-head on the same workload Stronger, but still not live
Shadow traffic Real OpenClaw traffic with side-by-side scoring Often enough for narrow claims
Narrow online rollout Operational decision support Best basis for public product claims

Comparison discipline

  • Use the same query set, scoring rubric, and prompt budget rules across all modes.
  • Keep the command, artifact path, and commit SHA for every published number.
  • Leave cells as TBD when the artifact does not exist yet.
  • Do not describe simulation-only results as real operator wins.

The canonical proof path is in docs/reproduce-eval.md.
