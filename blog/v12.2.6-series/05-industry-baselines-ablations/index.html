Baselines: what are we comparing against?

March 2026 | v12.2.6+ series

If OpenClawBrain is going to claim it's better, it needs to compare against methods people would actually use instead. Not straw men, not vague "AI memory" framing — real retrieval methods that a reasonable team might choose. This page leads with the numbers, then explains why each comparison method matters and where the evidence boundary sits.

The bottom line: the learned guide answers 97.5% of recorded-session queries correctly vs. 89.6% for the best search method — roughly 1 in 13 queries where search fails, the guide gets right. In the sparse-feedback test (10 seeds), the gap widens to 92.0% vs. 67.1%, using about 5x less context per query and winning 9 of 10 seeds head-to-head. These are benchmark results, not live rollout claims.

The controlled test: what happens when you remove each piece

Think of this like removing parts from a car to see which ones matter. Each row takes away a capability and shows what breaks.

MethodWhat it testsRight answers (of 4)
Keyword search The floor: searching by word similarity, no knowledge of how facts relate. Like using Google instead of asking a librarian. 0/4
Link-following Walking obvious connections between facts, without learning which matter most. Like following hyperlinks without judgement. 1/4
Map structure only Using the knowledge map's built-in relationships, before any learning. The "useful from day one" baseline. 2/4
Learned guide The same map after background learning has improved which paths the guide follows. 4/4

The learning curve tells the second half of the story. Map structure alone stays flat at 2/4 across all 16 rounds, while the learned guide reaches 4/4 by round 14. This is the core product story in concrete form: the knowledge map makes the guide useful from day one, and background learning makes it better over time. Search methods have no equivalent — they perform the same on day one and day one thousand.

Why these comparisons belong

QuestionWhy the answer matters
If keyword search were already enough, why build a memory layer? A memory product should beat a plain search floor, not just rename it.
If following links were enough, why learn anything? Simple link-following is a credible alternative. It needs to be tested head-to-head.
If map structure alone matched the learned guide, why keep the learning path? The product claim isn't just "there is a knowledge map." It's "the map helps immediately, and background learning makes it better over time."

What the comparison says honestly

  • The controlled test cleanly separates keyword search, link-following, map structure, and the learned guide.
  • Recorded-session replay: the guide scores 97.5% vs. 89.6% for the best search method on 800 queries.
  • Sparse-feedback (10 seeds) extends this to 18,000 total queries: 92.0% vs. 67.1% at about 5x less context.
  • We still do not have broad wins from real OpenClaw traffic. This is benchmark evidence, not live-traffic proof.

What stronger evidence looks like

LevelWhat it showsEnough for broad claims?

Shadow traffic (not yet) Real OpenClaw traffic with side-by-side scoring Often enough for narrow claims
Narrow live rollout (not yet) Serving real users with measurable outcomes Best basis for public product claims

Comparison discipline

  • Use the same query set, scoring rubric, and prompt budget rules across all methods.
  • Keep the command, artifact path, and commit SHA for every published number.
  • Leave cells as TBD when the artifact doesn't exist yet.
  • Don't describe the 4-query controlled test as a real-world win.
