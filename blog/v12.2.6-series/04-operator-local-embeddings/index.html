Post 4

Local-first operator rollout

March 2026 | v12.2.6+ series

OpenClawBrain should not require a complicated remote stack before it becomes useful. The default operating shape is local-first: local embedder, local async teacher, fast daemon query path, and background training that can start after first use.

Default stack

  • local embedder: BAAI/bge-large-en-v1.5
  • local async teacher: Ollama qwen3.5:35b
  • runtime path: openclawbrain serve plus the OpenClaw adapter or hook pack

Use it first

WORKSPACE=~/.openclaw/workspace
BRAIN_DIR=~/.openclawbrain/main

mkdir -p "$BRAIN_DIR"
openclawbrain init --workspace "$WORKSPACE" --output "$BRAIN_DIR"
openclawbrain serve --state "$BRAIN_DIR/state.json"

python3 -m openclawbrain.openclaw_adapter.query_brain \
  "$BRAIN_DIR/state.json" \
  "what are the current operating rules?" \
  --chat-id smoke-test \
  --format prompt \
  --exclude-bootstrap

This is the point of the product direction: you can start using the brain before replay, harvest, or teacher labeling has finished.

Train it in the background

openclawbrain replay \
  --state "$BRAIN_DIR/state.json" \
  --sessions ~/.openclaw/agents/main/sessions \
  --fast-learning \
  --resume \
  --checkpoint "$BRAIN_DIR/replay_checkpoint.json"

openclawbrain harvest \
  --state "$BRAIN_DIR/state.json" \
  --events "$BRAIN_DIR/learning_events.jsonl" \
  --tasks split,merge,prune,connect,scale \
  --json

The scanner/harvester layer and the async teacher both belong here, not in the live request path.

What should stay off the hot path

  1. teacher labeling
  2. historical replay
  3. structural graph surgery
  4. anything that turns a live query into a long workflow job

Why this local-first shape matters

  • privacy and lower operational friction for the default deployment story
  • fast first-use path for OpenClaw operators
  • a clearer product claim: the live value comes from the learned runtime router, not from an inline teacher call
