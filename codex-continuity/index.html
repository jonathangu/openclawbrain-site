Codex continuity tutorial

Codex does the deep work. Telegram keeps you connected.

OpenClawBrain turns OpenClaw into a mobile operator layer around Codex UI. Codex stays the high-bandwidth workbench at the computer; OpenClaw and Telegram give you quiet status, progress checks, watched completion, blockers, failures, and handoff briefs while you are away.

Telegram is not a second coding UI. It is the operator surface.

The workflow

Jonathan now does much of his deep coding in Codex UI. That works beautifully when he is physically at the machine, but it is awkward from a phone. OpenClaw agents already live on the same computer and can be reached through Telegram.

OpenClawBrain connects those worlds. It reads local Codex state, decides what is worth surfacing, and sends only operator-grade summaries through OpenClaw.

Text these to your OpenClaw agent

These commands are intentionally small. They are for awareness and handoff, not noisy coding telemetry.

/brain codex status
/brain codex threads
/brain codex threads openclawbrain
/brain codex watch --latest
/brain codex watch <thread-id>
/brain codex handoff
/brain codex handoff <thread-id>

Status pulse

/brain codex status answers: source, visible thread count, active goal count, watch count, latest thread, current goal status, and whether writes are disabled.

Find the right work

/brain codex threads lists recent threads. Add a filter like openclawbrain, publish, or resume to find the relevant goal faster.

Watch while away

/brain codex watch --latest watches the latest visible Codex thread. Use /brain codex watch <thread-id> when you want a specific thread.

Return-to-computer brief

/brain codex handoff gives observed facts, Codex-reported claims, evidence, and next actions. Add a thread id for a specific handoff.

Cool progress-check workflows

Leaving the computer

Text /brain codex watch --latest. OpenClawBrain will stay quiet unless Codex completes, fails, blocks, needs approval, or hits auth trouble.

Checking from your phone

Text /brain codex status for the fast pulse: what Codex appears to be working on and whether the bridge is reading fresh state or stale-labeled fallback.

Finding a buried goal

Text /brain codex threads memory or /brain codex threads decagon to filter recent Codex threads by title, workspace, or visible goal objective.

Walking back to the machine

Text /brain codex handoff. The brief tells you what is observed, what Codex merely claimed, what evidence exists, and what to do next.

When a notification arrives

A watch notification should mean a terminal or attention-worthy event happened: completion, failure, blocker, approval-needed, or auth-failure.

Testing the safety rail

/brain codex goal ... and /brain codex steer ... refuse by default. That is intentional until Telegram-to-Codex writes are explicitly enabled with guardrails.

How to read the reply

sqlite_fallbackThe bridge is reading local Codex state in read-only fallback mode.
read-only/stale-labeledTreat the status as awareness, not proof. Ask for a handoff or verify in Codex UI before acting.

Writes: disabledTelegram can inspect/watch/handoff, but it will not remote-control Codex by default.

The bridge is deliberately quiet. A phone notification should mean something changed, something finished, or a decision is needed.

Why OpenClawBrain belongs in the middle

This is not just a notification bridge. The hard part is deciding which local state has authority in the current user context.

Continuity

It remembers that Codex UI is the deep workbench and Telegram is the low-bandwidth operator surface.

Authority

It can treat an old workflow, preference, or handoff as useful history without letting it silently override the current instruction.

Boundaries

It stores redacted watch and audit events, not durable raw Codex telemetry.

Upgradeability

The bridge is OpenClawBrain-owned, so Jonathan's personal OpenClaw checkout can stay stock and update with upstream.

Safety defaults

Read-only status, threads, watches, and handoffs are the default. Telegram-to-Codex write control stays disabled unless explicitly feature-flagged later with trusted sender checks, repo allowlists, risk classification, and confirmation.

default posture:
  read local Codex state
  summarize only meaningful terminal events
  store redacted audit/watch events
  never write through SQLite
  never patch OpenClaw core
  keep Telegram-to-Codex writes disabled

Install or upgrade

openclaw plugins install clawhub:openclawbrain@0.2.29 --force
openclaw plugins enable openclawbrain
openclaw gateway restart

After install, ask from Telegram or OpenClaw: /brain codex status. The bridge should answer with a quiet operator summary, not a stream of Codex internals.
