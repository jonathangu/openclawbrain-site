OpenClawBrain troubleshooting

When the lane looks wrong, keep the scope tight and compare the right fields.

The recovery loop is boring on purpose: keep the same selected home, rerun install, restart the gateway, compare detailed status, and only then decide whether proof is actually the problem.

A fresh sterile published host can still show STATUS warn even while loadProof=status_probe_ready, serve state=serving_active_pack, and routeFn available=yes are already healthy. That is the difference between a noisy proof surface and a broken runtime.

Fast triage

Start with the signal that failed first. Do not change homes halfway through the diagnosis.

Load not proven

loadProof=status_probe_ready is missing

Rerun openclawbrain install --openclaw-home '~/.openclaw', restart the gateway, and compare detailed status again.

Headline warn

STATUS warn can still be a fresh-host state

If detailed status already shows the healthy runtime signals, the host may be fine even when proof still emits warning-level noise.

Serving state differs

serve state=serving_active_pack is missing

That is not automatically an install failure. A fresh or lightly used host can still be pre-promotion.

Route function missing

routeFn available=yes is missing

Check whether the runtime is already serving a promoted pack and compare with the proof page instead of assuming every host should match the exercised one.

Use the same compare surface: the healthy host returned STATUS ok together with the load proof, serving state, and learned route. If your host does not, the next step is usually reinstall plus restart, not a new theory.

Canonical recovery lane

When in doubt, rerun the public lane exactly. The restart command remains the only step without --openclaw-home. 

openclawbrain install --openclaw-home '~/.openclaw'
openclaw gateway restart
openclawbrain status --openclaw-home '~/.openclaw' --detailed
openclawbrain proof --openclaw-home '~/.openclaw'
  • 1. Install

    Repair the boundary

    Re-run install to converge the hook and pin the host again.

  • 2. Restart

    Reload the live runtime

    Restart makes the gateway pick up the updated state.

  • 3. Verify

    Read status before proof

    Detailed status is the faster truth surface when proof is noisy.

Two mistakes to avoid

Scope drift

Changing the selected home mid-debug

If install, status, and proof do not point at the same home, you are no longer comparing the same boundary.

Package drift

Managing the plugin package without re-converging install

If you changed @openclawbrain/openclaw directly, rerun the install lane afterward so the hook and activation root converge again.
