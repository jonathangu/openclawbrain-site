\section{Abstract}
OpenClawBrain is a learned retrieval graph for LLM agents where workspace chunks are nodes and document transitions are weighted edges.
OpenClawBrain augments vector-seeded retrieval with trajectory-level learning: it updates edge confidence from full-path outcomes using corrected policy gradients.
It provides value on two axes: (1) retrieval accuracy through graph-structured traversal over co-occurrence edges between related chunks, and (2) context efficiency through trajectory-level learning.
In simulation, active context drops from 30 nodes on the first deployment query to 2.7 nodes on average in the last 10 queries (91\% reduction).
In production, on-demand routing supplies 3--13KB per query from 52--66KB baseline startup context.
The results in this paper are constrained to 15 deterministic simulations, 272 local tests, and the reproducible benchmark harness.
\section{1. The Problem}
Repeated workspace navigation can leak repeated context and amplify cost.
Most cost comes from one pattern: each query reloading too much workspace context.
A common failure mode is over-retrieval: each query re-injects a large fraction of workspace context, increasing latency and token cost.
This hurts retrieval quality as well as budget.
\textit{By "the graph is the prompt," we mean that traversal over the learned edge-weighted graph selects the subset of context for the expensive read stage.}
\paragraph{Objective.} Given a query $q$ and graph $G$, the traversal policy $\pi_W$ selects a candidate subset of nodes $F\subseteq V$ for expensive reads. We minimize expected context tokens with success-constrained control:
\[
\min_W \ \mathbb{E}_{q\sim\mathcal{D}}[\mathrm{tokens}(F)] \quad \text{s.t.} \quad \mathbb{E}_{q\sim\mathcal{D}}[\tilde z(q)]\ge\rho,\; |F|\le B.
\]
Let $\tilde z\in\{0,1\}$ denote success and define the signed outcome $z=2\tilde z-1\in\{-1,+1\}$.
In practice, this objective is implemented by signed-outcome updates with baseline control.
In a deploy flow, \textit{check CI} $\to$ \textit{inspect manifest} $\to$ \textit{rollback} $\to$ \textit{verify}, similarity retrieval groups all four steps as related, not as ordered.
It also fails on conflict: when both “run tests before deploy” and “skip tests for hotfix” are close in vector space, RAG can still surface the wrong instruction.
OpenClawBrain augments seeding (vector search or heuristics) with learned traversal; it is not a standalone retrieval method without an entry mechanism.
\section{2. The Graph}
OpenClawBrain stores memory as a directed graph.
Nodes hold chunks; edges represent possible next reads.

\subsection*{2.1 Traverse API}
The router interface is explicit:
\[
\texttt{route\_fn(query, candidate\_ids)}\to\text{subset of } candidate\_ids.
\]
It may use query-local cues and is optional in low-cost routes.
\begingroup\scriptsize
\begin{verbatim}
def _tier(weight, config):
    if weight <= config.inhibitory_threshold:
        return "inhibitory"
    if weight >= config.reflex_threshold:
        return "reflex"
    if config.habitual_range[0] <= weight < config.habitual_range[1]:
        return "habitual"
    return "dormant"
def traverse(graph, seeds, config, route_query=None):
    route_fn = getattr(config, "route_fn", None)
    frontier = sorted(seeds, key=lambda s: s[1], reverse=True)[: config.beam_width]
    frontier = [(node_id, score) for node_id, score in frontier if graph.get_node(node_id)]
    seen = {node_id: 1 for node_id, _ in frontier}
    used_edges = {}
    fired = [node_id for node_id, _ in frontier]
    steps = []
    for _ in range(config.max_hops):
        raw = []
        suppressed = set()
        for source_id, source_score in frontier:
            for target_node, edge in graph.outgoing(source_id):
                tier = _tier(edge.weight, config)
                if tier == "dormant":
                    continue
                if tier == "inhibitory":
                    suppressed.add(target_node.id)
                    continue
                use_count = used_edges.get((source_id, target_node.id), 0)
                effective_weight = edge.weight * (config.edge_damping ** use_count)
                if effective_weight <= 0:
                    continue
                raw.append((source_id, target_node.id, source_score * effective_weight, tier))
        if suppressed:
            raw = [r for r in raw if r[1] not in suppressed]
        if not raw:
            break
        habitual = [r for r in raw if r[3] == "habitual"]
        reflexive = [r for r in raw if r[3] == "reflex"]
        selected = []
        if route_fn is not None and habitual:
            allowed = set(route_fn(route_query, [r[1] for r in habitual]))
            selected.extend(r for r in habitual if r[1] in allowed)
        else:
            selected.extend(sorted(habitual, key=lambda x: x[2], reverse=True))
        selected.extend(sorted(reflexive, key=lambda x: x[2], reverse=True))
        next_frontier = []
        seen_next = set()
        for source_id, target_id, score, tier in sorted(selected, reverse=True)[: config.beam_width]:
            if target_id in seen_next:
                continue
            if target_id not in seen:
                seen[target_id] = 1
                fired.append(target_id)
            else:
                seen[target_id] += 1
            used_edges[(source_id, target_id)] = used_edges.get((source_id, target_id), 0) + 1
            steps.append((source_id, target_id, tier, score))
            next_frontier.append((target_id, score))
            seen_next.add(target_id)
        frontier = next_frontier
    return TraversalResult(fired=fired, steps=steps, context="")
\end{verbatim}
\endgroup
Negative edges remain visible as inhibitory evidence and suppress wrong routes directly.
During traversal, inhibitory targets are removed from candidate expansion if they are connected by any edge with weight $\le -0.01$.
This is a hard-veto arbitration rule. It reduces recurrence of known-bad transitions and can increase false negatives with noisy outcomes.
\section{3. How It Learns}
Each query creates an episode with route state, action choices, and binary outcome.
Let $\tilde z\in\{0,1\}$ denote success and $z=2\tilde z-1\in\{-1,+1\}$.
\subsection*{3.1 Policy Gradients over Routes}
Let $\mathcal{A}(i)=\mathcal{N}(i)\cup\{\texttt{STOP}\}$ at node $i$, signed outcome $z$, and baseline $b$.
\[
\pi_W(a\mid s=i)=\frac{\exp\bigl((r_{ia}+w_{ia})/\tau\bigr)}{\sum_{j\in\mathcal{A}(i)}\exp\bigl((r_{ij}+w_{ij})/\tau\bigr)}
\]
\paragraph{Logits and derivation.}
Define logits
\[
\ell_{ij} \coloneqq \frac{r_{ij}+w_{ij}}{\tau},\qquad
Z_i=\log\sum_{k\in\mathcal{A}(i)}\exp(\ell_{ik}).
\]
\[
\log\pi_W(a\mid s=i)=\ell_{ia}-Z_i.
\]
For chosen action $a$:
\[
\frac{\partial \ell_{ia}}{\partial w_{ia}}=\frac{1}{\tau},\qquad
\frac{\partial Z_i}{\partial w_{ia}}=\frac{\pi_W(a\mid i)}{\tau}
\]
so
\[
\frac{\partial\log\pi_W(a\mid s=i)}{\partial w_{ia}}=\frac{1}{\tau}(1-\pi_W(a\mid i)).
\]
For any non-chosen action $j\neq a$:
\[
\frac{\partial \ell_{ia}}{\partial w_{ij}}=0,\qquad
\frac{\partial Z_i}{\partial w_{ij}}=\frac{\pi_W(j\mid i)}{\tau}
\]
so
\[
\frac{\partial\log\pi_W(a\mid s=i)}{\partial w_{ij}}=-\frac{\pi_W(j\mid i)}{\tau}.
\]
\[
\nabla_{\mathbf w_i}\log\pi_W(a\mid i)=\frac{1}{\tau}(\mathbf e_a-\boldsymbol{\pi}_i).
\]
In the unconstrained formulation, per-node updates sum to zero and redistribute mass within each node.
\subsection*{3.2 Why Myopic Updates Fail}
Myopic updates only credit the final edge before a terminal outcome:
\[
\Delta W\propto z\nabla_W\log\pi_W(a_t\mid s_t)
\]
That drops responsibility when early edges cause the final mistake.
The corrected estimator from \href{https://jonathangu.com/papers/rl-corrected-update-2016.pdf}{Gu (2016)} sums all steps in the path:
\[
\Delta W=\eta z\sum_{\ell=0}^{T}\nabla_W\log\pi_W(a_\ell\mid s_\ell)
\]
and with baseline and discount:
\[
\Delta W=\eta(z-b)\sum_{\ell=0}^{T}\gamma^\ell\nabla_W\log\pi_W(a_\ell\mid s_\ell)
\]
\subsection*{3.3 Full REINFORCE Update}
\[
\Delta w_{s_\ell, j} = \frac{\eta(z-b)\gamma^\ell}{\tau}\bigl(\mathbf{1}[j = a_\ell]-\pi_W(j\mid s_\ell)\bigr)
\]
This updates all outgoing edges at each visited node $s_\ell$.
Positive outcomes push mass toward the chosen edge and away from competitors; negative outcomes do the reverse. The updates sum to zero.
\[
\sum_{j\in\mathcal{A}(s_\ell)}\Delta w_{s_\ell, j}=0.
\]
\paragraph{Numerical example.}
Consider a node with three outgoing edges plus STOP, all with $r=0$, current logits $(0.5, 0.3, -0.2, 0.0)$, $\tau=1$, and $\eta=0.1$.
\[
\pi(A\mid i)=0.342,\ \pi(B\mid i)=0.280,\ \pi(C\mid i)=0.170,\ \pi(\text{STOP}\mid i)=0.208.
\]
For $\gamma^0=1$, $b=0$, and $z=+1$ where $A$ is chosen:
\[
\Delta w_{iA}=+0.066,\ \Delta w_{iB}=-0.028,\ \Delta w_{iC}=-0.017,\ \Delta w_{i\texttt{STOP}}=-0.021,\ \sum\Delta w=0.
\]
For $z=-1$:
\[
\Delta w_{iA}=-0.066,\ \Delta w_{iB}=+0.028,\ \Delta w_{iC}=+0.017,\ \Delta w_{i\texttt{STOP}}=+0.021.
\]
The heuristic updates only the chosen edge, whereas true PG updates all edges at the node.

\begin{verbatim}
def _softmax_probs(graph, node_id, temperature):
    import math
    outgoing = graph.outgoing(node_id)  # [(target_node, edge), ...]
    logits = {}
    for target_node, edge in outgoing:
        target_id = getattr(target_node, "id", target_node)
        relevance = float(edge.metadata.get("relevance", 0.0))
        logits[target_id] = (relevance + edge.weight) / temperature
    logits["STOP"] = 0.0
    max_logit = max(logits.values())
    exp_sum = sum(math.exp(v - max_logit) for v in logits.values())
    return {k: math.exp(v - max_logit) / exp_sum for k, v in logits.items()}


def apply_outcome_pg(graph, fired_nodes, outcome, config=None, baseline=0.0):
    config = LearningConfig() if config is None else config
    updates = {}
    z = 2.0 * float(outcome) - 1.0
    advantage = z - baseline
    if len(fired_nodes) < 2:
        return updates
    for step_idx in range(len(fired_nodes) - 1):
        source_id = fired_nodes[step_idx]
        chosen = fired_nodes[step_idx + 1]
        probs = _softmax_probs(graph, source_id, config.temperature)
        scalar = config.learning_rate * (config.discount ** step_idx) * advantage / config.temperature
        for target_node, edge in graph.outgoing(source_id):
            target_id = getattr(target_node, "id", target_node)
            is_chosen = 1.0 if target_id == chosen else 0.0
            delta = scalar * (is_chosen - probs.get(target_id, 0.0))
            edge.weight = _clip_weight(edge.weight + delta, config.weight_bounds)
            graph._edges[source_id][target_id] = edge
            updates[f"{source_id}->{target_id}"] = delta
    return updates
\end{verbatim}

\subsection*{3.5 Why Not AlphaGo Search}
AlphaGo uses value-function and tree-search budgets.
OpenClawBrain receives one real query outcome per turn and must stay cheap.
Corrected policy-gradient credits over full trajectories fits this cost model: learn from real outcomes, not from millions of simulated futures.

\subsection*{4.2 Decay (Dampened Dormancy)}
Unused edges decay geometrically.
\[
\tilde{w}=w\cdot\left(\frac12\right)^{\Delta q / h}
\]
where $\Delta q$ is queries since last traversal and $h$ is half-life.
\subsection*{4.3 Edge Damping}
Within one trajectory, repeated traversals of the same directed edge lose immediate weight:
\[
\tilde{w}=w\cdot\lambda^k
\]
with $k$ repeats in this episode.
\subsection*{4.4 Update Rule}
\begingroup\scriptsize
\begin{verbatim}
def apply_decay(graph, config=DecayConfig(half_life=80)):
    for source_id in list(graph._edges.keys()):
        for target_id, edge in list(graph._edges[source_id].items()):
            edge.weight = _clip_weight(edge.weight * (0.5 ** (1 / config.half_life)), config.weight_bounds)
            graph._edges[source_id][target_id] = edge

\subsection*{5.2 Self-Regulation}

Three homeostatic mechanisms maintain graph health without manual tuning.

\paragraph{Tier hysteresis.} The habitual band was widened from $[0.2, 0.6)$ to $[0.15, 0.6)$. This buffer prevents edges near the dormant/habitual boundary from thrashing between tiers due to stochastic update noise.

\paragraph{Homeostatic decay.} Rather than fixing the decay half-life, the maintenance loop adjusts it adaptively. The controller monitors the reflex edge ratio (fraction of edges with $w \ge 0.6$) and targets a band of 5--15\%. If the ratio exceeds 15\%, the half-life decreases by 3\% (faster decay). If below 5\%, it increases by 3\% (slower decay). The half-life is bounded to $[60, 300]$ cycles.

This is analogous to firing-rate homeostasis in neuroscience: the system regulates a behavioral outcome (reflex proportion) rather than individual parameters.

\paragraph{Synaptic scaling.} A soft per-node outgoing weight budget of 5.0 prevents hub domination. For each node with total positive outgoing mass $L_1 > 5.0$, weights are scaled by $(5.0 / L_1)^{0.25}$ (fourth-root scaling). This preserves relative edge rankings while capping absolute influence, analogous to synaptic scaling in biological neural networks.

\subsection*{6.2 Learning Dynamics: Policy Gradient vs Heuristic}
\textbf{Claim.} Under realistic mixed feedback and concept drift, the policy-gradient update is quantitatively superior to the heuristic update.

\textbf{Setup.} This uses the full v2 protocol: 50 nodes in 3 topic clusters sharing a central hub, 3 query types with a designated correct path, and feedback mix 65\% positive, 20\% negative, 15\% partial (first 2 hops correct, remainder wrong). Concept drift occurs at step 200 as target paths rotate to different clusters. Ten seeds run 400 steps, and both update rules start from identical graph states.

The key metric is \emph{branch separation}: the weight of the correct outgoing edge at the hub minus the weight of the strongest wrong outgoing edge.

\section{8. Limitations and Conclusion}
\subsection*{Limitations}
Cold start remains expensive, because the bootstrap graph begins highly exploratory and requires deliberation until enough feedback is collected.
Embedding quality is a hard dependency: poor chunking or weak vectors produce weak seeds and weak routes.
Results are from 15 deterministic simulation runs and production deployment across three production brains (1,160 + 584 + 286 nodes) with real OpenAI embeddings, 698 replayed sessions, and 259 injected learning corrections. Benchmark claims in this paper are limited to reproducible artifacts in this repository.
External benchmarks are now included (MultiHop-RAG and HotPotQA distractor), in addition to repository-internal simulation and retrieval analyses.
Seeding quality is a hard dependency. If the correct node is never seeded (due to poor embeddings or out-of-vocabulary queries), OpenClawBrain cannot learn a route to it. In production, we mitigate this with embedding model quality (OpenAI text-embedding-3-small) and keyword fallback seeding. We have not systematically characterized the failure rate under degraded seeding, which remains an open evaluation gap.
Homeostatic decay targets (5--15\% reflex ratio) are calibrated on three production brains and may need adjustment for substantially different workloads.
Self-regulation parameters (decay bounds, scaling budget, hysteresis bands) are hand-tuned on current production graphs and have not been validated across diverse deployment scenarios.
The self\_learn API depends on agents correctly identifying their own failures; false self-corrections could degrade retrieval quality.
Not evaluated:
\begin{itemize}
  \item Across diverse domains/users beyond our production deployment
  \item Long-horizon catastrophic forgetting in multi-month workloads
  \item Head-to-head with standard RAG baselines under identical corpora
\end{itemize}
Corrected policy-gradient updates are better than myopic credit assignment, but they still depend heavily on discount tuning.
Small graphs under 10 files show limited benefit and can look similar to direct retrieval.
Damping is fixed at 0.3 in these experiments and is not adapted per graph.

\subsection*{When NOT to use OpenClawBrain}
Prefer a vector DB for simple static documentation where one-off reads dominate and adaptation does not amortize.
Avoid OpenClawBrain for single-query tasks with little recurring structure.
Avoid OpenClawBrain when you require fine-grained, continuous feedback during generation; the default update is one outcome label per traversal.

\subsection*{Conclusion}
\begingroup\small
OpenClawBrain improves retrieval accuracy through graph-structured traversal (+7.7\% on MultiHop-RAG, +33.6\% on HotPotQA versus embedding-only baselines) and compresses context on repeated tasks through trajectory-level learning (30\(\to\)2.7 nodes in simulation, 52--66KB\(\to\)3--13KB in production).
The true REINFORCE policy gradient achieves 13--19\texttimes{} better branch separation under mixed feedback (0.0052--0.0058 vs 0.0003--0.0004), recovers 244\texttimes{} faster after concept drift (0.1 vs 24.4 steps), and produces 30\% less weight inflation (+38.2 vs +54.1 total weight delta) compared to the heuristic update.
Homeostatic decay, synaptic scaling, and tier hysteresis maintain stable graph dynamics without manual tuning.
The \texttt{self\_learn} API enables autonomous agent correction and reinforcement without human feedback, closing the outcome observation loop.
These benefits require graph structure (co-occurrence edges between related chunks) and, for compression, repeated query patterns.
On diverse, non-repeating queries, learning provides no benefit and can degrade accuracy.
Install: \texttt{pip install openclawbrain}. The core package has no required third-party runtime dependencies beyond Python; embeddings and LLM generation are provided through callback interfaces or optional extras.
Code: \url{https://github.com/jonathangu/openclawbrain}
\endgroup

\appendix
\section{Deployment and Operations}
\subsection*{5.4 Migration and Session Replay}
New machines or revisions rerun bootstrap and optionally replay session logs for weight warm-start.

Session replay is a warm-start path for a new graph.
The system reads historical session logs, extracts user queries, and runs each through the graph to seed, traverse, and apply outcome updates.
No LLM is required for replay to restore traversal structure.
Replay can warm-start graph structure before live traffic when the graph lacks history.
v9.3.0 adds incremental replay via \texttt{last\_replayed\_ts}: only new sessions since the last replay are processed, avoiding redundant traversals.
The system also extracts assistant responses via \texttt{extract\_interactions}, enabling outcome-weighted replay where corrections receive negative feedback and acknowledged queries receive positive feedback.

\subsection*{5.5 Learning Database Integration}
OpenClawBrain nodes are not limited to file chunks. The OpenClaw integration injects learning database entries --- corrections, directives, and teachings harvested from human feedback --- as first-class graph nodes with real embeddings.
Each learning node carries its error class, prevention rule, recurrence count, and source text. CORRECTION nodes create inhibitory edges, while TEACHING and DIRECTIVE nodes add positive knowledge without inhibitory edges. This means the graph can retrieve not just workspace content but also \textit{how to avoid past mistakes}.
In production, 259 active learning entries from three agents are embedded and queryable alongside workspace chunks.

\subsection*{5.6 Operational Tooling}
\texttt{openclawbrain doctor} validates state integrity: Python version, state file validity, node/edge counts, embedder metadata, dimension consistency, and journal writability.
\texttt{openclawbrain info} reports graph statistics: node count, edge count, embedder name/dim, tier distribution percentages, and state file size.
A per-brain journal system logs queries, learning events, and health snapshots for operational observability.

\subsection*{5.7 Persistent Daemon (openclawbrain daemon)}
OpenClawBrain ships a long-running NDJSON worker for production.
The daemon keeps both \texttt{state.json} and \texttt{VectorIndex} hot in memory, reducing startup/teardown overhead from subprocess-per-query execution.
In practice this removes repeated file/graph load and index rebuild costs, often saving 100--800ms per query in large brains.
\paragraph{Protocol}
Transport is NDJSON over stdin/stdout.
Each request line is:
\begin{verbatim}
{"id":"<id>","method":"query|learn|maintain|health|info|save|reload|shutdown","params":{...}}
\end{verbatim}
Each response line is either:
\begin{verbatim}
{"id":"<id>","result":{...}}
{"id":"<id>","error":"..."}
\end{verbatim}
The daemon is stateless per request and stateful in process memory.
\paragraph{Methods}
\begin{itemize}
  \item \texttt{query} performs retrieval/traversal and returns fired context.
  \item \texttt{learn} applies outcome feedback to the graph.
  \item \texttt{maintain} runs health, decay, merge, prune, and compact steps.
  \item \texttt{health} returns health metrics.
  \item \texttt{info} returns graph, embedder, and runtime metadata.
  \item \texttt{save} persists in-memory state to disk.
  \item \texttt{reload} reloads persistent state and index into memory.
  \item \texttt{shutdown} flushes and exits.
\end{itemize}
\paragraph{Production timing}
On a Mac Mini M4 Pro with OpenAI embeddings, daemon mode reports 397ms embed + 107ms traverse, for 504ms total per query.
This compares to 600--1300ms when run with per-call subprocess execution.
\paragraph{Durability and shutdown}
Auto-save uses a configurable interval in daemon settings, and \texttt{SIGTERM}/\texttt{SIGINT} trigger graceful shutdown with state persistence.
\paragraph{Integration with OpenClaw}
The daemon is designed as the backend for TypeScript/Node.js agent frameworks such as OpenClaw.
The framework runs it as a child process and sends NDJSON requests over stdin, keeping the Python runtime hot and isolated.

\subsection*{5.8 Production Lessons}
\begingroup\scriptsize
Across multiple months of production operation, the following lessons surfaced:
\begin{enumerate}
\item \textbf{Embedder metadata corruption.} The CLI save path silently overwrote OpenAI embedder metadata with hash-v1 defaults. Every learn or replay command degraded the state. Fix: \texttt{save\_state} now checks the meta dict for embedder info before falling back to defaults (v9.3.1).
\item \textbf{Orphaned correction nodes.} Injecting learning corrections as graph nodes without connecting them to workspace nodes produced orphans unreachable by traversal. Fix: connect each correction to its top-3 most similar workspace nodes via cosine similarity (v9.2.0).
\item \textbf{Batch overflow.} Sending all texts to OpenAI in a single embedding call hit the 300K token API limit at 1,635 texts. Fix: chunk into batches of 100 (v9.3.0).
\item \textbf{Context savings measurement (Axis 2).} Before OpenClawBrain,\\
agent session startup loaded 52--66KB of workspace files.\\
After migration, on-demand routing supplies 3--13KB per query,\\
and the reduction came from moving reference material\\
(checklists, error class maps, design guides) into the graph\\
while keeping safety rules always-injected.
\item \textbf{Recalibrated tier thresholds.} Production telemetry across three brains (5,863 edges total)\\
moved defaults to
$\texttt{reflex\_threshold}\allowbreak=0.6$,
$\texttt{habitual\_range}\allowbreak=0.2\ldots 0.6$, and
$\texttt{inhibitory\_threshold}\allowbreak=-0.01$ so inhibitory weights near $-0.035$ now suppress downstream traversal.
\item \textbf{Inhibitory edges require matching dimensions.} The inject module validates that new vectors match the index dimension. Hash embeddings (1024-dim) cannot create meaningful connections in an OpenAI index (1536-dim). This is now a hard error.
\item \textbf{Live correction pipeline.} Corrections reach the graph within 2 hours: harvester ingests from sessions, LLM classifies, SQLite stores, then \texttt{inject\_batch} embeds new nodes, connects them, and creates inhibitory edges. No manual rebuild required.
\item \textbf{Real-time correction flow.} \texttt{query\_brain.py}\allowbreak\texttt{ --chat-id} persists fired nodes per conversation, enabling same-turn \texttt{learn\_correction.py} flows. A content-hash ledger and batch harvester prevent duplicate injections. This moved end-to-end insertion from fixed 2-hour batch windows toward same-turn correction when needed.
\end{enumerate}
\endgroup

\subsection*{5.8 Real-time Correction}
\begingroup\scriptsize
The real-time pipeline adds per-conversation persisted fires and immediate correction hooks.
\texttt{query\_brain.py --chat-id} stores fired nodes in \texttt{fired\_log.jsonl} with rolling retention.
\texttt{learn\_correction.py} now applies outcomes directly against logged paths using \texttt{apply\_outcome\_pg}, while a background batch harvester still supports delayed ingestion.
This avoids duplicate repair through a content-hash ledger and preserves the same correction semantics as manual rebuild-free workflows.
\texttt{learn\_correction.py} shortens practical latency from 2-hour batch updates toward same-turn correction in active sessions.
\begin{enumerate}
  \item Fired-node persistence is per conversation.
  \item Same-turn injection is supported via \texttt{learn\_correction.py}.
  \item Batch harvester deduplicates by content hash ledger.
  \item Latency path moves from 2-hour-only to same-turn when invoked in flow.
\end{enumerate}
\endgroup

\subsection*{5.10 Two-Timescale Architecture}
OpenClawBrain separates online adaptation from structural maintenance.
\paragraph{Online learning (fast loop).}
Each query executes \texttt{traverse()} with existing edge weights, logs the trace, receives outcome feedback, and applies per-query updates through \texttt{apply\_outcome()} or \texttt{apply\_outcome\_pg()}.
\[
\texttt{query}\rightarrow\texttt{traverse}\rightarrow\texttt{log\_trace}\rightarrow\texttt{feedback}\rightarrow\texttt{learn}
\]
\paragraph{Maintenance (slow loop).}
Structural upkeep is periodic and asynchronous to requests.\\
\texttt{run\_maintenance()} is scheduler-agnostic and composes health, decay, merge, prune,\\
and compact stages in a single entry point.
\paragraph{Operational worker model.}
Online updates and maintenance are available through the daemon API methods, so production services keep state resident, process boundaries stable, and request latency predictable.
\[
\texttt{health}\rightarrow\texttt{decay}\rightarrow\texttt{merge}\rightarrow\texttt{prune}\rightarrow\texttt{compact}
\]
\paragraph{Constitutional anchors.}
Node metadata assigns maintenance authority:
\begin{itemize}
  \item \texttt{constitutional}: never decay, never prune, never merge.
  \item \texttt{canonical}: half-life decay at 2x scale, merges only with explicit confirmation.
  \item \texttt{overlay} (default): normal maintenance behavior.
\end{itemize}
\paragraph{Context lifecycle.}
Operational context flow is:
\[
\texttt{Files}\rightarrow\texttt{Sync}\rightarrow\texttt{Graph}\rightarrow\texttt{Maintain}\rightarrow\texttt{Compact}
\]
