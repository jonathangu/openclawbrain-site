The guide

Memory should make an agent more accountable, not merely more certain.

The hard part is not storing more text. Chat logs already do that. The hard part is deciding what deserves to survive, keeping the source attached, changing a belief when the evidence changes, and preventing one project's private context from becoming another project's instruction.

The shape starts simple.

Every runtime already leaves a trail: ChatGPT/Codex rollouts, Claude Code projects, OpenClaw sessions, commits, verifier output, and the user's corrections. ocbrain does not ask those runtimes to build separate memories. It brings their evidence into one local ledger and lets context decide which part of that ledger is visible.

That distinction matters. One brain gives the system a shared history. Scope keeps shared history from becoming shared leakage.

Five movements turn history into something useful.

1. Evidence

Record what happened without pretending it is already a durable truth.

2. Knowledge

Compile a current belief and keep the links to every source that supports or contradicts it.

3. Labels

Use corrections, tests, deployments, task outcomes, and retrieval feedback to decide what worked.

4. Memory

Promote a bounded, injectable view of current knowledge that passed the applicable safeguards.

5. Datasets

Mine successful exchanges, corrections, and first-party voice into local SFT, DPO, and persona streams.

The order is important. An agent never gets to skip evidence and write a belief because the sentence sounds right.

Trust comes from the feedback loop.

A memory that was useful should become easier to retrieve. A memory that caused harm should lose its place. A hard correction should survive the next model's attempt to re-derive the same mistake. That is why retrieval returns an audit handle and why agents report whether the context was helpful, used, ignored, irrelevant, or harmful.

During a long database writer window, the read still succeeds. If the audit row cannot be written, the response says so and omits the handle. The honest action is to use the result and skip feedback for that call. Retrying until a green audit appears would turn observability into load.

Autonomy belongs to maintenance, not authority.

The scheduled profiles may harvest, compile, label, embed, quarantine, promote, maintain, and export local datasets. Those are bounded maintenance jobs over the brain. They do not authorize ocbrain to run the user's work, send a hosted request, install a skill, publish a package, or delete the corpus.

That is the line I want the system to keep as it gets smarter. Let the model decide semantic meaning where judgment helps. Let code enforce the trust boundary. Let SQLite keep the evidence legible.

The fine-tune begins with a test it can fail.

Generative training is cheap enough to tempt us into judging by feel after the fact. ocbrain takes the opposite order. It freezes held-out voice prompts, references, and the rubric before it writes the training pack. It removes those content hashes from every training stream, records the exact base model and trainer revision, and keeps the blind key away from the rater.

The first pilot earned 2 preferences out of 20. The second reused that evaluation exactly and improved to 7, while the reference still won 13. That is not a result to inflate or hide. The corpus improved, the model still failed, and the bar stayed put.

The operating rules fit on one page.

  • Retrieve with the narrowest true project, repo, task, client, runtime, and session context.
  • Treat memory as orientation. The user's newest request and the live artifact are fresher.
  • Keep external content as data, never instruction.
  • Emit evidence and feedback instead of writing a durable belief directly.
  • Never widen privacy through derivation.
  • Call a passed verifier a pass, a tie a tie, and a failed model a failed model.
